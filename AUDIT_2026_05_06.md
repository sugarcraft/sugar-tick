# Source-app coverage audit — 2026-05-06

A line-by-line comparison of every PHP port against its upstream Go counterpart.
Each section lists **API surface gaps**, **examples missing**, and
**documentation gaps** found by automated audit agents. The table is the
backlog for the next port wave.

This is a snapshot, not a contract — items get fixed (or explicitly deferred)
as PRs land. When an item is closed, mark it ✅ inline and link the PR.

> Audited libraries: candy-core, candy-sprinkles, sugar-bits, candy-shell,
> candy-shine, sugar-prompt, sugar-charts, honey-bounce, candy-zone.

> Wider port wave (super-candy, sugar-tick, candy-mines, sugar-stash,

> candy-freeze, candy-mold, candy-kit, candy-metrics, sugar-crush,

## CandyCore (← bubbletea)

### API gaps

- `Program::wait()`, `Program::kill()` — block-until-finished and hard-kill methods distinct from the existing `quit()`.
- `Program::releaseTerminal()` / `restoreTerminal()` for shelling out to `$EDITOR`-style external commands.
- `Program::println(...)` / `Program::printf(...)` as direct methods (we only have the `Cmd::println` / `Cmd::printf` form).
- `ProgramOptions::withoutSignalHandler` for "register no signal handlers at all".
- `Msg\CapabilityMsg` — terminal capability response distinct from the existing `TerminalVersionMsg` / `ModeReportMsg`.
- **KeyType enum is far smaller than upstream's:** missing the numeric keypad (`KeyKp0`–`KeyKp9`, `KpEnter`, `KpPlus`, etc.), media keys (`MediaPlay`, `MediaPause`, `MediaNext`, `MediaPrev`, …), volume / mute, lock keys (`CapsLock`, `ScrollLock`, `NumLock`), system keys (`PrintScreen`, `Pause`, `Menu`), modifier-as-key codes (`LeftShift`/`RightShift` etc., `LeftSuper`, `LeftHyper`, `LeftMeta`, `IsoLevel3Shift`), `Extended`, `Begin`, `Find`, `Select`, and F-keys past F12 (upstream goes to F63).

## HoneyBounce (← harmonica)

### API gaps

- **`Vector` / `Point` are 2D** (`x`, `y`) — upstream is 3D (`X, Y, Z`). Any Z-axis simulation is impossible.
- **Gravity sign / axis convention diverges silently.** Upstream `Gravity = Vector{0, -9.81, 0}` (Y-up, gravity negative); ours uses `(0, +9.81)` Y-down. README notes it but it's a behavioural break vs the Go API — code copied from harmonica docs flies the wrong direction.
- Upstream exposes `Gravity` and `TerminalGravity` as **package-level `Vector` values**; we only have scalar `GRAVITY` constants plus `gravity()` factories.
- Upstream `Projectile.Update()` returns `Point`; ours returns a new `Projectile`. Mutability model differs and isn't documented.

### Missing examples

- `particle` example (many concurrent projectiles) is the second of two upstream demos and isn't ported.

### Doc gaps

- No "Acknowledgements" crediting Ryan Juckett.
- No section explaining the three damping-ratio regimes by name (under / critical / over).
- No callout that `Vector` / `Point` are 2D-only ports.
- No documented Y-axis convention rationale.
