- **[gotcha]** GitHub Actions matrix fan-out is NOT a free speedup — each matrix cell pays the full runner-boot + `actions/checkout@v5 fetch-depth: 0` cost (~30-60s for a monorepo with thousands of commits), so a job that took 30s serial-per-lib can take LONGER as a matrix because every cell re-clones the full history. Before splitting a serial loop into `strategy.matrix`, time one iteration: if per-lib work < ~45s, the per-cell checkout overhead dominates and matrix fan-out is a wash or regression. The `sync-sugarcraft.yml` parallelization (commit ae017e75) had to be reverted for this reason. Alternatives that DON'T pay per-cell checkout: GNU Parallel inside a single job (`echo "$LIBS" | xargs -P 5 -I{} ./sync-one.sh {}`), or a shallow `fetch-depth: 50` if the script only needs recent history.
- **[fix]** When the user says "undo the last commit and re-push" after a workflow change regression, the safe sequence is `git reset --hard HEAD~1 && git push --force-with-lease origin master` ONLY if you have explicit user permission — `--force-with-lease` is gentler than `--force` (refuses if upstream moved) but still rewrites public history. Confirm before running; never force-push master autonomously. Per CLAUDE.md: force-pushes to main/master are explicitly listed as destructive and require user authorization for each instance.
- **[gotcha]** `awalsh128/cache-apt-pkgs-action@v1` does NOT execute apt's post-install scripts on cache restore by default. ffmpeg's transitive `libblas` dep installs `/usr/lib/x86_64-linux-gnu/libblas.so.3` via `update-alternatives` in its post-install script — without it, ffmpeg crashes at runtime with `ffmpeg: error while loading shared libraries: libblas.so.3: cannot open shared object file`. - **[gotcha]** PHP FFI's `\FFI::cdef()` eagerly resolves ALL declared symbols at library load time — declaring a function the target libc doesn't export (e.g., POSIX 2024 `tcsetwinsize`/`tcgetwinsize` on glibc < 2.36, which is most non-bleeding-edge Linux distros) causes the entire FFI handle to fail with `Failed resolving C function '<name>'`, breaking ALL subsequent FFI calls — not just the missing one. Fix: build the cdef block per-platform — gate POSIX-2024-or-later / Darwin-only / Linux-only declarations behind `if (PHP_OS_FAMILY === 'Darwin')` (or equivalent) when concatenating the cdef string. Canonical pattern: `candy-pty/src/Libc.php::cdef()`. Bare cross-platform declarations only work for symbols guaranteed in every supported libc back to the oldest CI target.
- **[gotcha]** macOS arm64 + PHP FFI + variadic libc functions = ABI mismatch. Real libc `ioctl(int, unsigned long, ...)` is variadic, and the arm64 variadic ABI puts varargs on the stack while fixed args sit in `x0`–`x7`. A fixed-arg PHP FFI cdef (`int ioctl(int fd, unsigned long request, void *arg)`) pushes the third arg into `x2` while the kernel reads it from the stack → ioctl returns `-1` with no useful errno. Workaround: route through non-variadic POSIX 2024 wrappers (`tcsetwinsize`/`tcgetwinsize` for TIOCSWINSZ/TIOCGWINSZ) on Darwin only. PHP FFI cannot reliably express variadic signatures — declaring `int ioctl(int, unsigned long, ...)` breaks Linux too because FFI rejects `FFI\CData` arrays passed as variadic params. There is no portable PHP-FFI-only fix for arbitrary variadic libc calls on macOS arm64.

- **[gotcha]** Path-repo `composer.lock` mismatch on PR branches is EXPECTED, not a bug. `candy-core/composer.lock` (and any other consumer's lock) pins `sugarcraft/candy-pty` to `"version": "dev-master"`. When CI checks out a PR branch where the dep's HEAD is at `ai/<branch>`, composer reads the symlinked path-repo's branch as `dev-ai/<branch>` and rejects the lock as incompatible — yielding a 7-cell failure pattern (Test PHP 8.3/8.4 · PHPStan 8.3/8.4 · macOS · Windows × consumer-lib) on the PR. Do NOT regenerate the lock on the PR branch — that just shifts the failure to the post-merge master push. The pattern resolves automatically once the PR is merged and the dep's HEAD is back on master. The non-consumer matrix cells (e.g. candy-wish for a candy-pty change) ARE the canary — if they're green, the underlying fix works; merge the PR and the consumer cells go green on the next master run.

- **[gotcha]** In `candy-pty` pump tests that spawn a child to assert output bytes, prefer a single-write child like `printf 'TOKEN\n'` over multi-write `echo A; echo B` — `echo` chains issue two separate `write()`s, and the pump's `STDIN_EOF_GRACE_SEC` deadline can trip between them, truncating the cassette / pump-output to just the first token. The flake reproduces ~1-in-3 even on a fast host. Single-write `printf` is race-free against the EOF grace window. Coverage of the multi-write path already lives in `candy-pty/tests/Posix/PosixPumpEofGraceTest.php`, which uses `sleep`-padded loops specifically to exercise that timing — don't duplicate it from happy-path tests.

- **[gotcha]** PHP fatal `Could not check compatibility between <FakeChildClass>::method(): <ConcreteStub> and <ParentInterface>::method(): <InterfaceType>, because class <ConcreteStub> is not available` fires when a test stub class declares a covariant return type pointing at ANOTHER class defined later in the same file. PHP's compatibility check runs at class-load time and cannot see classes that haven't been declared yet — even though both end up in the same file. Fix: type stub method signatures against the INTERFACE (`public function master(): \SugarCraft\Pty\Contract\MasterPty`), not the concrete sibling stub class (`public function master(): FakeMasterPty`). Co-locating multiple stub classes in one test file is fine; just don't cross-reference them in return/parameter types — reference the shared interface instead. Triggered while writing `tests/PtyPoolTest.php` where `FakePtyPair::master()` returned `FakeMasterPty` declared later in the same file.
- **[gotcha]** When writing a test stub `class Fake<X> implements <Interface>`, the method signatures must match the INTERFACE byte-for-byte including parameter names, default values, and nullable markers — a stub for `MasterPty::read(int $len = 8192, ?float $timeout = null): ?string` written as `read(int $maxBytes, float $timeoutSec = 0.0): ?string` will pass syntax but fail at class-load with a `Declaration of ... must be compatible with ...` warning that PHPUnit's `failOnWarning=true` upgrades to a fatal. Same for `size(): array` — the interface's `@return array{cols: int, rows: int, xpix: int, ypix: int}` PHPDoc is informational, but every consumer test that asserts on the array shape will break if the stub returns a 2-key subset. Always copy-paste the interface signature when writing a stub; don't paraphrase.

- **[gotcha]** PTY pump teardown race: when a fast child writes a small payload (e.g. `printf 'session-0-line\n'`) and exits before the pump's first `tick()`, naively marking the session done as soon as `$child->exited()` returns true causes those bytes to be lost — they sit in the master buffer unread because the session was removed from the live set before the next `stream_select` round. Fix: watermark the moment the child is first observed exited (`$session->childExitedAt = microtime(true)`), then keep the session live until either master returns 0 bytes (EOF — `read()` returns `''`) OR `now - childExitedAt > flushDeadlineSec` (e.g. 0.5–1.0s). Reading master EOF immediately wins; the deadline is the cap. Canonical pattern: `candy-pty/src/Posix/MultiPump.php::tick()`. Same trap applies any time a PTY consumer checks `child->exited()` AND drains the master in the same loop — always drain past the exit signal.
- **[pattern]** PtyPool / FFI-handle pool gating: when designing any "pool the expensive thing" wrapper around a candy-pty / candy-* primitive, measure the per-call cost FIRST before committing to a pool architecture. The libc cdef is cached as a per-process static in `Libc::lib()` (so handle reuse cost ≈ 0.002ms/call, cold-load ≈ 0.008ms/call, full `PosixPtySystem::open` ≈ 0.05ms/call) — no per-open FFI cost is left to amortize. PTY pairs are 1:1 with their slave path and can't be re-bound. The pool's value is then BOOKKEEPING (typed `PoolExhaustedException` at capacity, idempotent `release()`, `drain()` on shutdown) — NOT handle reuse. Profile before scoping; if the cost is dust, scope the pool down to back-pressure + accounting only.
