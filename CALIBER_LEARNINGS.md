# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** Tests that assert env-var defaulting behavior (e.g. `Session::fromEnvironment()` falling back to `SSH_CLIENT` when `SSH_CONNECTION` is missing) must clear BOTH `$_SERVER` AND the process env. Clearing only `$_SERVER[$key]` leaves `getenv($key)` returning the host process's value (e.g. `USER=my` on the dev box). Pattern: snapshot `getenv($k)` for each key in `setUp()`, `unset($_SERVER[$k]); putenv($k);` to clear both, then restore both in `tearDown()`.
- **[gotcha]** A YAML comment-stripper that only handles `whitespace + #...` (`/\s+#.*$/`) leaves bare comment-only lines (`# top-level comment` with no leading whitespace) intact. Use `/^\s*(?:#.*)?$/` for the "skip blank/comment" check so both indented and column-0 comment lines are dropped.
- **[gotcha]** `git add sugar-wishlist/` will sweep up vim swap files like `sugar-wishlist/.README.md.swp` if the editor was open while you staged. Either close the editor before staging, add `.*.swp` to a `.gitignore`, or stage explicit subdirs.
- **[correction]** Adding a new sibling library to the monorepo requires updating THREE separate places, not two: root `composer.json` (require + repositories), `.github/workflows/ci.yml` matrix `lib:` list, AND `README.md` (library count, the table of libraries, and the test-loop shell snippet). Missing the CI matrix entry means the new lib's PHPUnit suite silently never runs in PRs. **Why:** ci.yml is a hand-maintained list, not glob-driven. **How to apply:** after every `composer require sugarcraft/<lib>` registration, grep for the previous lib name in `.github/workflows/ci.yml` and `README.md` and add the new one alongside it.
- **[correction]** Phase 10 ("Polish & public launch — SugarBuzz") in `CONVERSION.md` is NOT complete just because the per-library code/tests are merged — it explicitly includes the `sugarbuzz.dev` website (`docs/` skeleton with palette + library grid + quickstart), VHS-recorded README GIFs, and a per-library README/composer.json polish pass. **Why:** marking a multi-deliverable phase complete on partial work hides outstanding scope. **How to apply:** before marking any phase "done" in TodoWrite, re-read that phase's section in `CONVERSION.md` end-to-end and check every `- [ ]` item.
- **[pattern]** After registering a new lib in root `composer.json`, run `composer update sugarcraft/<lib> --quiet && composer show sugarcraft/<lib>` to verify it resolves through the path repository — `composer install` won't refresh the lock for a newly-added path package, only `composer update <name>` will.
- **[gotcha]** `.github/workflows/vhs.yml` has a hand-maintained matrix `lib:` list (parallel to ci.yml). When adding `.vhs/*.tape` files to a library that isn't in the vhs.yml matrix, the workflow silently does NOT render any GIFs for it — `find . -name '*.gif' -path '*/.vhs/*' | wc -l` will show the count short, but `gh run view <id> --json jobs` shows ALL listed jobs as `success`. After adding tapes for a new lib, grep `.github/workflows/vhs.yml` for an existing lib name and add the new one alongside.
- **[pattern]** Public-site lib detail pages (`docs/lib/<slug>.html`) reference per-lib GIFs via `https://raw.githubusercontent.com/sugarcraft/<lib>/main/.vhs/<n>.gif` (the synced org repo's `main` branch), NOT a relative `.vhs/` path. The monorepo path is `<lib>/.vhs/<n>.gif` and that's where READMEs reference them. Don't accidentally use the monorepo-relative path in `docs/`.
- **[gotcha]** `echo` and `exit` are PHP language constructs, NOT functions — so you cannot define `function echo(...)` (parse error) and you cannot call `\exit(main($argv))` (runtime `Call to undefined function exit()`). Both forms compile-/run-fail and cannot be "namespaced". When porting older code or using these in `match` arms (`'help' => echo(...)`), wrap the output in a real helper named `say()`/`out()` and use bare `exit(main($argv))` (no leading backslash) at the bootstrap line. Run `php -l <file>` on every CLI bin you touch — these failures point at the wrong line if you only spot-check at runtime.
- **[correction]** Monorepo-wide contributor docs (locale codes, naming policy, conversion roadmap, etc.) belong at the REPO ROOT alongside `MATCHUPS.md`/`CONVERSION.md`/`PROJECT_NAMES.md`, NOT under a specific lib's directory like `candy-core/lang/`. **Why:** every lib uses these codes when contributors add `lang/<locale>.php` files, and burying the doc inside `candy-core/` leaks contributor-only guidance into the downstream-distributed `sugarcraft/candy-core` Packagist package. **How to apply:** when creating a new cross-cut reference doc, place it at the root and add it to `AGENTS.md`'s "Source-of-truth files" list so future agents discover it.
- **[gotcha]** From a `.claude/worktrees/<name>/` worktree, `git checkout master` fails with `fatal: 'master' is already used by worktree at '/home/sites/sugarcraft'` because the primary worktree owns master. Don't try to switch — just `git fetch origin master --quiet && git reset --hard origin/master` to update the current detached HEAD, then `git checkout -b claude/<feature>` to start a fresh branch. Or work in a non-master branch from the start.
- **[pattern]** When patching `T::translate`'s lookup chain (e.g. adding base-language fallback `fr-fr` → `fr` → `en`), add three regression tests in lockstep: (1) regional locale falls back to base when no regional file exists, (2) regional file takes precedence over base when both exist, (3) regional locale falls all the way through to `en` when neither regional nor base file exists. Without all three, future refactors of the `??` chain can silently break ordering.
- **[pattern:project]** Plans in `plans/x-*.md` are drafted ahead of implementation and routinely cite APIs that don't actually exist in candy-core (e.g. `x-editor.md` referenced `ExecRequest::run(static fn() => Editor::edit(...))`, but `Cmd::exec` only accepts `string|array $command + ?\Closure $onComplete` — no closure-as-command variant). When the planned shape doesn't match reality, document the deviation in a `## Discovered during execution` section in the plan file itself before adapting — keeps the plan honest for future readers and explains in-place why the implementation diverges from the snippet they're reading.
- **[gotcha]** Tests that assert env-var defaulting behavior must clear THREE places in `setUp()`: `putenv($key)`, `unset($_ENV[$key])`, AND `unset($_SERVER[$key])`. Clearing only one or two leaks the host process's value into `getenv()` / `$_ENV` / `$_SERVER` lookups (e.g. the dev box's `USER=my`, `EDITOR=vim`). Pattern: snapshot `getenv($k)` per key in `setUp()`, then `putenv($k); unset($_ENV[$k], $_SERVER[$k]);` to clear all three; in `tearDown()` restore via `putenv("$k=$saved")` when the snapshot was a string, or `putenv($k)` (no `=`) to unset when it was `false`. Canonical in `candy-core/tests/Util/EditorTest.php` and `sugar-bits/tests/TextArea/TextAreaEditorTest.php`.

Either close the editor before staging, add `.*.swp` to a `.gitignore`, or stage explicit subdirs (`sugar-wishlist/src sugar-wishlist/tests sugar-wishlist/README.md ...`) instead of the whole directory.
- **[gotcha]** After `gh pr create` / `gh pr merge --delete-branch`, the script prints `Warning: N uncommitted changes` (any small N) even on a clean tree — this is `gh` complaining about the lockfile diff that `composer install` left behind, or untracked sibling files like `plans/` from a different branch's work, not a real uncommitted-source warning. Don't react to it; verify with `git status` if unsure.

- **[gotcha]** `composer validate --strict` flags every `"sugarcraft/*": "@dev"` entry as `unbound version constraints (@dev) should be avoided` — EXPECTED in this monorepo and the correct shape for path-repository sibling deps pre-1.0. Drop `--strict` or filter the unbound-`@dev` warnings when scripting validation.
- **[convention]** All per-library `composer.json` files share the same metadata block: `keywords` (lowercase kebab + always include `"sugarcraft"` and the upstream Go project name), `homepage: "https://github.com/detain/SugarCraft"`, single author (`Joe Huss <detain@interserver.net>`, role `Maintainer`), and `support` block with `issues` / `source` / `docs`. Insert these BEFORE `require`, AFTER `license`, to match established ordering.
- **[pattern]** When auditing per-library files (READMEs, composer keys, demos) across all libs, drive the loop from a hard-coded list of the 20 canonical slugs (as of 2026-05): `for d in candy-core candy-sprinkles honey-bounce candy-zone sugar-bits sugar-charts sugar-prompt candy-shell candy-shine candy-kit candy-freeze sugar-glow sugar-spark candy-wish sugar-wishlist candy-metrics candy-mold candy-tetris super-candy sugar-crush; do ...; done`. `wc -c < $d/README.md` flags missing/empty files immediately; glob/find won't reveal an empty file is conceptually missing.

- **[gotcha]** A `for d in ...; cd /home/sites/SugarCraft/$d; vendor/bin/phpunit` loop without subshell parens leaks the working directory between iterations, and any `cd` that fails silently leaves you running the next command in the wrong place — output for libs after the first often comes back empty. Wrap each iteration body in `(cd "$d" && ...)` so the cwd resets per-loop.
- **[gotcha]** The Bash tool's working directory persists across separate Bash calls. After `cd /home/sites/SugarCraft/sugar-charts && cmd`, the NEXT Bash call still starts in sugar-charts. Always use absolute paths or prefix every Bash call with `cd /home/sites/SugarCraft && ...` to anchor it; if you get a mysterious empty response from `ls`/`cat`, suspect the cwd before suspecting the file.
