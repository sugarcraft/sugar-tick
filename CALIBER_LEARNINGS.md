# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** Real-PTY tests MUST call `\stream_set_blocking($pair->master()->stream(), false)` before any read on the master — without it, `fread` on the master fd ignores the `$timeout` argument and blocks the whole PHP process until the kernel buffer has data. Symptom: a test that should fail with `ExpectTimeoutException` instead hangs forever.
- **[gotcha]** macOS arm64 + PHP FFI + variadic libc functions = ABI mismatch. Real libc `ioctl(int, unsigned long, ...)` is variadic, and a fixed-arg PHP FFI cdef pushes the third arg into `x2` while the kernel reads it from the stack → ioctl returns `-1` with no useful errno. Workaround on Darwin: route through non-variadic POSIX 2024 wrappers (`tcsetwinsize`/`tcgetwinsize`) or shell out to `stty -f /dev/fd/<fd>`.
- **[gotcha]** Bash watchdog patterns that `cd` into a lib subdir AND launch `( sleep N && pkill ... ) &` BEFORE the cd command lose CWD context — the watchdog subshell forks early and the foreground `vendor/bin/phpunit` invocation runs in the parent's CWD, not the lib's. Even the form `cd /path/to/lib && ( sleep N && pkill ... ) > log 2>&1 &` is unsafe — the `&` backgrounds the WHOLE compound (cd included), so the foreground phpunit on the next line still runs in the parent CWD with exit 127 `No such file or directory`. SAFEST fix: use the lib's absolute path for phpunit (`/home/sites/sugarcraft/<lib>/vendor/bin/phpunit`) AND anchor every other phpunit-dependent path (cassette tmp file, --bootstrap, --configuration) with absolute paths.
- **[gotcha]** Tracking "has this float been initialized?" with `$prevField > 0.0` is fragile when the field is computed from `round(microtime(true) - $startTime, 3)` — the first event in the first millisecond of a recording legitimately produces 0.0, and the guard then mistakes it for "never set". Use a separate `bool $hasPriorEvent = false` flag instead. Symptom from this anti-pattern: the second event's gap-detection silently skips its first invocation, breaking any `tRaw`/idle-trim arithmetic that depends on it. Canonical fix: `candy-vcr/src/Recorder.php::writeEvent()`.
- **[gotcha]** Real-PTY tests that spawn a child with a `sleep N` between visible outputs MUST give `RecordCommand` a non-EOFing stdin — passing `\fopen('/dev/null', 'r')` EOFs immediately, and `PumpOptions::stdinEofGraceSec` (default 300ms) closes the master before the child wakes from sleep. Symptom: cassette contains only the pre-sleep bytes; the post-sleep `printf 'post\n'` is never recorded. Fix: `[$r, $w] = \stream_socket_pair(STREAM_PF_UNIX, STREAM_SOCK_STREAM, STREAM_IPPROTO_IP); $cmd = new RecordCommand($r);` and keep both ends open for the duration of the test. Close both in the `finally`.
- **[pattern]** Host TTY safety-net pattern for any CLI command that drops the host stdin into raw mode: install `register_shutdown_function([self::class, 'rescueRestore'])` + `pcntl_signal(SIGTERM/SIGHUP, [self::class, 'handleRescueSignal'])` with `pcntl_async_signals(true)`, hold the Termios snapshot in a STATIC class property (closures can't survive past the run() frame), and drop a `sys_get_temp_dir() . '/<cmd>-rescue.<pid>'` marker file containing the host TTY path (`posix_ttyname(STDIN)`) so an external `stty sane < /path/to/tty` can recover from SIGKILL. Make handlers signal-safe (no allocation, no logging) and idempotent. Canonical: `candy-vcr/src/Cli/RecordCommand.php::installRescueHandlers()`.
- **[pattern]** Testing static-handler safety nets that re-raise signals: fork via `pcntl_fork()`, prime the static snapshot in the child, invoke the handler, and have the test-stub Termios append a marker line to a tempfile on `restore()` BEFORE the re-raise kills the child. The parent reaps via `pcntl_waitpid` and asserts `pcntl_wifsignaled($status) && pcntl_wtermsig($status) === SIGTERM` plus the marker file contains the restore evidence but NOT the trailing "REACHED_AFTER_HANDLER" sentinel. Canonical: `candy-vcr/tests/Cli/RecordCommandSafetyNetTest.php`.
- **[gotcha]** When testing static-property class state across multiple PHPUnit tests, reset all static properties in `tearDown()` via Reflection — test order is not guaranteed to be source order, and a leaked snapshot from one test will cause the next test's "no snapshot installed" assertion to fail mysteriously. `$ref->getProperty('foo')->setAccessible(true); $prop->setValue(null, $default);`.
- **[pattern]** PTY-real PHPUnit tests skip cleanly on FFI-less / sandboxed CI via a 5-line `requirePtySyscalls()` preamble: `PHP_OS_FAMILY === 'Windows'`, `extension_loaded('ffi')`, `is_readable('/dev/ptmx') && is_writable('/dev/ptmx')`, `extension_loaded('pcntl')` (only when spawning with `controllingTerminal: true`), and `is_executable('/bin/echo')` (or whichever child binary you spawn). For interactive children like vim, also probe `command -v vim` as a fallback to the hard-coded `/usr/bin/vim` path so the test gracefully skips on stripped CI images.
- **[convention]** `candy-vt`'s `SugarCraft\Vt\Screen\Screen` exposes dimensions as PUBLIC READONLY PROPERTIES — `$screen->cols` and `$screen->rows`, NOT method calls. Calling `$screen->cols()` errors with `Call to undefined method`. This breaks the bare-named-accessor mental model used elsewhere (`Terminal::screen()`, `Terminal::cursor()` are methods); only the dimension scalars on the readonly `Screen` snapshot are properties. Same pattern on `Buffer` (`$buf->cols`, `$buf->rows`).
- **[pattern]** Driving vim in a deterministic PHPUnit integration test: spawn with `-N -u NONE -i NONE <scratch>` (nocompatible, skip vimrc, no viminfo writeback) and pre-load the entire keystroke script onto a `stream_socket_pair()` write-end BEFORE invoking `RecordCommand::run()` — `\fwrite($stdinWrite, "iHello\x1b:wq\r")`. Keep BOTH socket ends open until after `run()` returns; closing the write end early EOFs the pump. Accept either exit code 0 or 1 from vim — `:wq` returns 1 on some builds when the buffer is unchanged. Canonical: `candy-vcr/tests/Integration/ShirleyVimTest.php`.
- **[pattern]** Cleaner watchdog cleanup: capture `WATCHDOG_PID=$!` right after the backgrounded `( sleep N && pkill ... ) > /tmp/wd.log 2>&1 &`, run the foreground command (saving `EXIT=$?`), then `kill $WATCHDOG_PID 2>/dev/null` BEFORE inspecting the output. Without that final kill, the watchdog stays alive and may kill an unrelated `vendor/bin/phpunit` invocation N seconds later in the same shell session.
- **[gotcha]** `fprintf($stream, $sprintfBuiltString)` is a trap when the second arg was built via `sprintf` and contains literal `%` characters (e.g. left over from `%%` → `%` percent-sign escapes in a `%.2f%%` template). `fprintf` re-interprets the percent signs as format specifiers and throws `ArgumentCountError: N arguments are required, 2 given` at runtime. Fix: use `fwrite($stream, $alreadyFormattedString)` for any string that's already been through `sprintf` — `fprintf` is only safe when you pass the literal format string + the substitution args separately.
- **[pattern]** Recorder/pump overhead microbench protocol (PR P6.5.6 canonical): one untimed warmup iteration to seed FFI cdef + kernel pty allocator + composer autoload caches, then N=5 timed iterations per scenario, sort, take the median (not mean — robust against single-shot GC/preemption stalls), assert `(median_with - median_without) / median_without ≤ budget`. Local Linux noise floor on `bash -c 'seq 100000'` is ~0% (negative is normal — the recorder occasionally measures faster than the baseline due to scheduler luck); set the CI budget to 5% to absorb shared-runner jitter while still flagging the regression class this test exists for (serialization-per-chunk regressions land at dozens of percent). Canonical: `candy-vcr/tests/Integration/ShirleyOverheadTest.php`.
- **[fix]** When a merged PR leaves a stale `dev-ai/<old-branch>` pin on `sugarcraft/<dep>` inside a consuming lib's `composer.lock` — surfacing post-merge as CI "Interface SugarCraft\…\X not found" / class-load errors — fix on a NEW branch off master: (1) `cd <lib> && composer update sugarcraft/<dep>` to regenerate, (2) `sed -i 's|dev-ai/<new-branch>|dev-master|g' <lib>/composer.lock` to rewrite composer's new pin back to `dev-master`, (3) `composer validate` + full PHPUnit run, (4) commit & PR. Composer's `update` will always pin to the current branch name; the `sed` step is what makes the lock match the canonical `dev-master` shape the rest of the monorepo uses. Canonical fix: PR #487-followup on `ai/candy-vcr-lock-fix`.
- **[gotcha]** A class-not-found error in CI (e.g. `Interface "SugarCraft\Pty\Contract\Termios" not found`) on a freshly-merged-to-master commit, when the same suite passes locally, is almost always a stale `composer.lock` cross-lib pin pointing to a now-merged branch. Quick check: `grep -A1 '"name": "sugarcraft/<dep>"' <lib>/composer.lock` — if it shows `dev-ai/<old-branch>` not `dev-master`, that's the bug.
- **[gotcha]** `gh run view <id> --log-failed | grep -E '(FAIL|Error|Tests:)'` often returns empty even when the run failed loudly — the log lines are prefixed with `<job-name>\t<step-name>\t<timestamp>` and PHPUnit failures don't always contain those exact tokens (e.g. "An error occurred inside PHPUnit" + "Message:" pair). Fall back to `gh run view <id> --log-failed 2>&1 | tail -80` to actually see the failure text.
- **[gotcha]** `gh run list --workflow=ci.yml --branch=master --limit=1` returns the MOST RECENT run regardless of state — a freshly-pushed merge commit shows up as `queued`/`in_progress` with empty `conclusion` field. Filter via `--status=success --limit=1` to confirm CI is actually green, or use `--limit=12` and visually scan the conclusion column.
- **[gotcha]** `actions/cache@v5` with a `restore-keys:` fallback silently shadows freshly-committed updates to the cached file. When `.github/workflows/ci.yml` caches `<lib>/composer.lock` under `key: lock-cov-vN-<lib>-<hash>` with `restore-keys: lock-cov-vN-<lib>-`, a partial-match restore on the runner OVERWRITES the committed lock BEFORE `composer install` runs — so even a correct lock-pin fix in the repo (e.g. `dev-ai/<old> → dev-master`) does not land in CI. Symptom: PR fixes a class-not-found error locally and on the PR, but the post-merge master push fails identically. Fix: bump the `vN` suffix (`lock-cov-v2 → v3`) to invalidate the entire cache lineage. Vendor caches (`vendor-vN`, `vendor-cov-vN`, `vendor-stan-vN`, `vendor-win-vN`, `vendor-mac-vN`) live next to it and need bumping together — partial bumps leave shadowed vendor/ trees that re-resolve to the stale lock.
- **[convention]** Cache-key version-bump in `.github/workflows/ci.yml`: ALWAYS bump every `-vN-` key in lockstep (`lock-cov-vN`, `vendor-vN`, `vendor-cov-vN`, `vendor-stan-vN`, `vendor-win-vN`, `vendor-mac-vN`). Use `grep -n 'cov-v\|vendor-v\|key: lock-' .github/workflows/ci.yml` to enumerate them, then `sed -i 's/lock-cov-v2/lock-cov-v3/g; s/vendor-v2-/vendor-v3-/g; ...'` in one pass. Document the bump rationale in the inline comment block above `Cache composer.lock` so future bumps inherit the history.

- **[pattern:mutable-trait]** Standard immutable-with pattern via `SugarCraft\Core\Concerns\Mutable` trait: `use Mutable;` in the class, then `mutate(array $changes): static { return new static(...array_merge(get_object_vars($this), $changes)); }`. Works for simple cases where all properties are set via constructor and mutable fields are private. **Exception:** Libs with sentinel-bool needs (Style's `with()` method tracks `propsSet`; TextArea's `mutate()` has `validateSet`/`errSet`/`promptFuncSet` sentinels) must override `mutate()` in-class and document the exception. See `candy-core/src/Concerns/Mutable.php`, `candy-sprinkles/CALIBER_LEARNINGS.md` (style-with exception), `sugar-bits/CALIBER_LEARNINGS.md` (sentinel-bool pattern).
