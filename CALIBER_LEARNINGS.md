# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** The `Node.js 20 is deprecated... actions/download-artifact@v5 ... forced to run on Node.js 24` annotation on a failed workflow run is INFO-only, not the failure cause — it appears identically on green runs and on runs that died before spawning a job. Don't chase the Node 20 deprecation warning when debugging a real failure; look at the actual job logs (or the absence thereof, see the workflow-file-issue gotcha above) for the real signal.
- **[pattern]** Quick CI-config-touch lint combo before committing changes to `scripts/affected-libs.php` + `.github/workflows/ci.yml` (or `vhs.yml`): `php -l scripts/affected-libs.php && python3 -c "import yaml; yaml.safe_load(open('.github/workflows/ci.yml'))" && echo ci.yml: OK` — covers PHP syntax errors AND ci.yml YAML well-formedness in one shot. Catches both regressions cheaply before push; YAML parse errors in a workflow file silently fail the workflow on GitHub with no useful surface error.
- **[gotcha]** GitHub Actions `strategy.matrix` with multiple top-level keys produces a CARTESIAN PRODUCT, not parallel/zipped lists. Declaring `matrix: { lib: [46 items], vhs_version: [46 copies], ttyd_version: [46 copies] }` spawns 46×46×46 ≈ 97k matrix cells, far past the 256-job-per-run limit, and the run fails at parse time with `0s` duration and `total_count: 0` jobs (looks identical to a YAML syntax error from `gh run view`). For per-cell parallel values that don't actually vary across the matrix, source them from `needs.<job>.outputs.*` in step bodies/keys, NOT from the matrix. For paired tuples that vary together, use `matrix.include:` with explicit `{lib: X, vhs_version: Y}` entries — never parallel top-level arrays.
- **[gotcha]** Per-matrix-job `git diff HEAD~1 HEAD` checks fail silently when the job's `actions/checkout@v5` uses default `fetch-depth: 1` — there's no HEAD~1 commit to diff against, so the `grep -qE "^${{ matrix.lib }}/(src|examples|bin|\.vhs)/"` returns nonzero for every lib and every job emits `needs_render=false`. Either pin the matrix-job checkout to `fetch-depth: 2`, or — better — delegate the should-render decision to an upstream `needs.<job>.outputs.libs` array (the upstream job has the depth and the logic) and have each matrix cell test membership via `jq -e --arg lib "$LIB" 'index($lib) != null' <<<"$LIBS_JSON"`. Never re-derive change detection from a shallow checkout.
- **[pattern]** When the working tree picks up unrelated edits from Caliber's pre-commit refresh (e.g. `CALIBER_LEARNINGS.md` modified at the start of a session) and you want a clean diff for the actual fix, `git stash push -- CALIBER_LEARNINGS.md` isolates the unrelated edits, then `git diff --stat HEAD` shows only the real change-set. Pop the stash back before commit so Caliber's edits land with the fix in a single ship-as-you-go PR.
- **[convention]** Six libs in the monorepo have ZERO `require[sugarcraft/*]` deps — `candy-hermit`, `sugar-calendar`, `sugar-readline`, `sugar-stickers`, `sugar-table`, `sugar-toast`. A `candy-core` change does NOT fan out to them, and they only appear in the affected-libs matrix when their own files change or a force-all trigger fires.
- **[gotcha]** When validating a `scripts/affected-libs.php`-style PHP CLI tool that writes JSON to stdout and debug to stderr, test it across these scenarios at minimum: `--all` (forces all), a `candy-core` file (max fan-out), a leaf-only lib file (`sugar-crush/src/Game.php` → 1 affected), a mid-graph lib (`candy-sprinkles`), a force-all trigger (`composer.json`, `.github/workflows/ci.yml`), and a no-op file (`README.md` → empty matrix). The empty-matrix case is the trickiest: the JSON must emit an empty array, not omit the key.
- **[gotcha]** Verifying `scripts/affected-libs.php` scoped behaviour with `--range HEAD~1..HEAD` is unreliable — if the last commit touched any force-all trigger (`.github/workflows/ci.yml`, root `composer.json`/`composer.lock`, or `scripts/affected-libs.php` itself) the script reports `force_all=yes, changed=0, affected=46/46` and you NEVER exercise the reverse-dep walk. Always inspect `git diff --name-only <range>` before drawing conclusions.
- **[pattern]** VHS CI workflow auto-discovers tapes within a registered lib — `.github/workflows/vhs.yml` renders EVERY `.vhs/*.tape` in the lib via `for tape in .vhs/*.tape; do vhs "$tape"; done`, gated by `hashFiles(format('{0}/.vhs/*.tape', matrix.lib)) != ''`. Adding a new tape to an already-registered lib needs NO workflow changes — just commit the new `.tape`.
- **[gotcha]** The Bash tool's working directory persists across separate Bash calls. After `cd /home/sites/sugarcraft/<lib> && cmd`, the NEXT Bash call still starts in `<lib>`. Always anchor with `cd /home/sites/sugarcraft && ...` or `cd /home/sites/sugarcraft/<lib> && ...`. Same trap bites HEREDOC-written `/tmp/dbg.php` scripts: `require_once __DIR__ . '/vendor/autoload.php'` resolves `__DIR__` to `/tmp/`, NOT to the working dir of the heredoc-emitting Bash call. Use absolute autoload paths in any standalone PHP script you stage under `/tmp/`.
- **[gotcha]** `gh pr create` prints `Warning: 1 uncommitted change` to stderr but STILL creates the PR successfully and returns 0 — the warning is informational. The PR will not contain that uncommitted file. Recovery: after `gh pr create` returns the URL, check `git status` for staged-but-not-committed files; if any belong with the change-set, follow up with a separate commit + push BEFORE attempting `gh pr merge`. Common cause: Caliber's pre-commit-hook refresh touches `<lib>/CALIBER_LEARNINGS.md` AFTER your `git add` set was finalised.
- **[convention]** When shipping a v1 milestone PR for a SugarCraft lib, the 256-square candy-themed icon (`media/<slug>.png` + `docs/img/icons/<slug>.png`) CANNOT be generated by the agent — image creation is outside the tool surface. Call out the missing icon explicitly in the PR body under "Plan deviations".
- **[gotcha]** Public classes in this monorepo are `final` by project convention. PHPUnit tests CANNOT mock or wrap them via an anonymous `new class($args) extends FooMiddleware { … }` — PHP fatals with `cannot extend final class`. Use a sibling `implements` of the same interface as a probe/observer instead.
- **[gotcha]** A green `vendor/bin/phpunit` run (`OK (N tests, M assertions)`) HIDES PHP runtime warnings unless you pass `--display-warnings --display-deprecations`. A single `W` character mixed into the dot progress is the only signal. Always re-run with `vendor/bin/phpunit --display-warnings --display-deprecations 2>&1 | tail -200` after any "OK" run that touched rendering / array-key code paths before declaring victory.
- **[gotcha]** Destructured locals like `[$tl, $tr, $bl, $br, $h, $v] = $this->getStyleChars();` are scoped to ONE method — a refactor that splits rendering across sibling private methods silently leaves variables undefined in the new methods. PHPUnit reports `OK` because PHP's undefined-variable coercion-to-`null` plus implicit `null . "\n"` produces a string that's "close enough" to the snapshot. Re-destructure at the top of every render method that needs the style chars.
- **[gotcha]** PHP `%` on a float operand silently emits a warning under `--display-warnings` (e.g. `($tick / 20) % ($totalWidth + 20)` — division produces float even when both operands are int). Use `intdiv($a, $b) % $mod` for integer modulo arithmetic, or cast `(int)`.
- **[gotcha]** Array-bounds guards like `if ($index < count($arr))` are HALF a check — a negative `$index` still slips through and triggers `Undefined array key -5` warnings. Always guard both ends: `if ($index >= 0 && $index < count($arr))`. Plain `vendor/bin/phpunit` won't flag it because the array access just returns `null`; `--display-warnings` is the only signal.
- **[pattern]** When a single source-file change introduces a new public class (e.g. extracting `PartitionSegment` out of `Partition.php` for PSR-4 compliance), the new file's namespace MUST match its directory. PSR-4 autoload-from-classmap will silently NOT find the class if the namespace is wrong. Run `composer dump-autoload -o` after extracting bundled classes.
- **[fix]** When building dependency-aware CI gating (the `scripts/affected-libs.php` graph), `require-dev` deps are explicitly EXCLUDED from the edge set — only `require[sugarcraft/*]` counts as a real edge.
- **[pattern]** When extracting bundled classes from a single source file, do the moves in this order: (1) `Write` the new per-class files first with the correct namespace, (2) `Edit` the original file with a single multi-block `old_string` that removes ALL the extracted definitions in one shot, (3) run `vendor/bin/phpunit` immediately to catch dangling `use` statements or missed references.
- **[gotcha]** `str_pad($line, $useWidth, ' ', STR_PAD_RIGHT)` counts BYTES, not visible columns, so any line containing ANSI escapes (`\x1b[38;2;…m…\x1b[0m`) gets under-padded by the byte length of the escapes. Fix: strip ANSI with `preg_replace('/\x1b\[[0-9;]*m/', '', $line)`, measure visible width with `mb_strlen($visible, 'UTF-8')`, then append `str_repeat(' ', max(0, $useWidth - $visibleWidth))` to the original (still-colored) line.
- **[gotcha]** A GitHub Actions workflow run that shows `failure` with duration `0s` and where `gh run view <id> --log-failed` returns `log not found` is a workflow-file/parse-error failure, NOT a job failure. `gh api repos/<owner>/<repo>/actions/runs/<id>/jobs` returns `{"total_count":0,"jobs":[]}` — no jobs were ever spawned. Recovery: don't keep digging via `gh` — lint the workflow YAML locally and audit `${{ }}` expressions, especially `fromJSON()`/`toJSON()` round-trips on `needs.<job>.outputs.*` and matrix-from-output wiring. A successful YAML parse doesn't rule out an invalid expression at runtime.
- **[gotcha]** Matrix expansion from an upstream job output requires a `toJSON()`/`fromJSON()` round-trip when the output value is a JSON array. The pattern `strategy.matrix.lib: ${{ fromJSON(needs.changed.outputs.libs) }}` only works if the upstream `changed` job assigns its output via `echo "libs=$(jq -c . file.json)" >> $GITHUB_OUTPUT` AND the consuming job uses `fromJSON()` to parse it back.

- **[gotcha]** PHPUnit can intermittently report a phantom `FAILURES! Tests: N, Failures: 1` summary on the same suite that returns `OK (N tests, M assertions)` on the next run. Recovery: re-run `vendor/bin/phpunit 2>&1 | tail -3` two or three times. If you get clean OK runs, treat the original failure as a flake.
- **[gotcha]** VHS render workflow uses `shivammathur/setup-php` with an explicit `extensions:` list — by default it installs `mbstring, intl, pcntl`. Libs that declare `ext-ssh2`, `ext-gd`, `ext-ffi`, `ext-pdo_sqlite`, `ext-sqlite3`, or other non-default extensions in their `composer.json` will fail at `composer install` in the VHS job with a missing-extension error. Audit per-lib extension requirements with `for f in */composer.json; do jq -r '.require // {} | to_entries[] | select(.key | startswith("ext-")) | .key' "$f"; done` and add the missing ones to `.github/workflows/vhs.yml`'s `extensions:` line (currently: `mbstring, intl, pcntl, ssh2`). `ext-json`/`ext-openssl` are part of PHP core/default builds and don't need adding.
- **[gotcha]** `gh run view --job <id> --log-failed` returns `error connecting to results-receiver.actions.githubusercontent.com` intermittently — it's transient GitHub infrastructure, not a permanent failure. Retry after 30-60s. The same call to `--json status,conclusion,jobs` continues to work because it hits a different endpoint.
- **[pattern]** `gh run watch <id> --exit-status` should be run in the background (`run_in_background: true`) rather than blocking foreground — render jobs can take 10+ minutes and a foreground call ties up the tool budget. Better: poll with `gh run view <id> --json status,conclusion,jobs --jq '{status, success: ([.jobs[]|select(.conclusion=="success")]|length), failure: ([.jobs[]|select(.conclusion=="failure")]|length), running: ([.jobs[]|select(.status!="completed")]|length)}'` every ~90s via `ScheduleWakeup` to check progress without blocking.
