# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** GitHub Actions: a `uses:` step that you want to gate a later step on (via `steps.<id>.outputs.cache-hit`) MUST have an explicit `id:` field. Without it, `steps.cache-ffmpeg.outputs.cache-hit` silently evaluates to empty-string (≠ 'true'), and the `if: steps.cache-ffmpeg.outputs.cache-hit != 'true'` fallback runs on EVERY job — even on cache hit. Symptom in `.github/workflows/vhs.yml`: "Install ffmpeg (fallback)" running ~15s of redundant apt work after a "Cache restored successfully" log line. Fix: add `id: cache-ffmpeg` to the `Cache ffmpeg` step (mirror for any other cache step gated this way).
- **[gotcha]** Unauthenticated `curl https://api.github.com/repos/.../releases/latest` from GitHub Actions runners is rate-limited to ~60 req/hr per runner-IP. On busy runners (or after a burst of `vhs.yml` runs) this hits HTTP 403, and even a 3-retry exponential-backoff loop exits failed — the `Resolve vhs / ttyd versions` step in `.github/workflows/vhs.yml` has been seen failing this way. Fix: use the authenticated 5000/hr limit — either `curl -H "Authorization: Bearer $GITHUB_TOKEN"` (with `env: GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}` on the step) or `gh api repos/.../releases/latest` (gh inherits the workflow token automatically).
- **[pattern]** When `git push` is rejected with `Updates were rejected because the remote contains work that you do not have locally` on a branch you've been pushing to, the cause is usually a workflow's auto-commit job — e.g. `vhs.yml`'s `commit` job pushes regenerated GIFs back to the branch. Recover with `git pull --rebase && git push`; do NOT force-push, you'll discard the generated artifacts. Always `git log --oneline -5` afterwards to confirm the rebase landed cleanly.
- **[gotcha]** `.claude/scheduled_tasks.lock` is a transient Claude Code scheduled-task subsystem file. It routinely appears as untracked in `git status` and is the cause of `gh pr create`'s `Warning: 1 uncommitted change` when no other working-tree edits exist. Don't commit it; don't waste time investigating. If it's the lone untracked file, the PR is complete.
- **[pattern]** Quick diagnostic for GitHub Actions cache effectiveness: `gh run view --job <id> --log | grep -E 'Cache (hit|miss|restored|saving|saved)'` — surfaces the `Cache hit for: <key>` / `Cache restored from key:` lines for every cache step in one pass without scrolling through tens of MB of log output.
- **[pattern]** To check whether the head commit's workflow run actually succeeded after a `gh pr merge`, query the auto-triggered master-push run directly: `gh run list --workflow=<name>.yml --limit 6` then `gh run view <id> --json status,conclusion,jobs --jq '{conclusion, total: (.jobs|length), failed: [.jobs[]|select(.conclusion=="failure")|.name]}'`. A failed `changed` job (the upstream that emits the matrix outputs) blocks every downstream matrix cell — fix it before chasing per-lib failures.
- **[pattern]** In GitHub Actions steps that fetch release metadata or assets, prefer the `gh` CLI over `curl | jq`: `gh release view --repo <owner>/<repo> --json tagName -q .tagName` for tags, `gh release download --repo <owner>/<repo> --pattern '<glob>' --output <path> --clobber` for asset downloads. Two wins over the curl approach: (1) `gh` auto-authenticates via `GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}` env so you get the 5000/hr limit for free, (2) `--pattern` collapses the curl-then-jq-then-curl asset-URL dance into a single command. Caveats: VHS asset names use `vhs_<ver>_Linux_x86_64.tar.gz` (underscores, capital L) — verify the exact pattern with `gh release view --repo charmbracelet/vhs --json assets -q '.assets[].name'` first.
- **[gotcha]** `awalsh128/cache-apt-pkgs-action@v1` does NOT execute apt's post-install scripts on cache restore by default. ffmpeg's transitive `libblas` dep installs `/usr/lib/x86_64-linux-gnu/libblas.so.3` via `update-alternatives` in its post-install script — without it, ffmpeg crashes at runtime with `ffmpeg: error while loading shared libraries: libblas.so.3: cannot open shared object file`. Fix: set `with: execute_install_scripts: true`. When changing this flag, bump the `version:` input (e.g. `1.0.0` → `2.0.0`) to invalidate existing caches that were built without the symlinks.
- **[gotcha]** When a follow-up workflow run is `in_progress` and you're trying to verify a fix, `gh run view --job <id> --log` returns `job <id> is still in progress; logs will be available when it is complete` — there is no streaming-log mode via gh. Either wait for the job to finish (via `ScheduleWakeup` to poll) or watch the UI directly. The user pasting a runtime error from a previous run's UI is faster than waiting for gh's log endpoint to become available.
- **[gotcha]** A `Verify tools` step that runs `ffmpeg -version | head -1` MASKS shared-library load failures: ffmpeg exits 127 with the libblas error on stderr, then `head -1` exits 0, and Bash's default (no `pipefail`) takes the last command's exit code — so the step is green even though ffmpeg is broken. Subsequent rendering steps fail with the real error far from the root cause. Fix: prepend `set -euo pipefail` to any `run: |` block that pipes a critical binary's output into `head`/`tail`/`grep`, so the upstream failure propagates.
- **[pattern]** Quick GitHub Actions workflow YAML lint without leaving the local shell: `python3 -c "import yaml; yaml.safe_load(open('.github/workflows/<file>.yml')); print('OK')"`. Catches structural/indent errors before pushing, but does NOT validate `${{ }}` expression semantics or action-input contracts — those still need a real run.
- **[gotcha]** `awalsh128/cache-apt-pkgs-action@v1` cannot reliably restore packages whose post-install scripts register `update-alternatives` symlinks (e.g. ffmpeg's transitive `libblas` dep, which creates `/usr/lib/x86_64-linux-gnu/libblas.so.3`). Even with `execute_install_scripts: true` the action's `dpkg --configure` invocation does NOT recreate update-alternatives entries — ffmpeg still crashes at runtime with `libblas.so.3: cannot open shared object file`. Compare working `candy-kit` (cache miss → fresh apt install) vs failing `candy-shine` (cache hit → broken libblas) in run 25825814166 for the diagnostic pattern. Fix: skip the cache action entirely for packages that touch update-alternatives — direct `sudo apt-get install -y --no-install-recommends <pkg>` on the runner-image's pre-indexed apt cache is ~15s and always runs real post-install scripts.
- **[pattern]** When comparing a working-vs-failing job pair to localize a CI issue, fetch both job IDs from the user-provided URLs and diff their cache/install logs side by side: `for jid in <good> <bad>; do echo "==== $jid ===="; gh run view --job $jid --log 2>&1 | grep -iE 'cache (hit|miss|restored)|installing|installed|<pkg-name>'; done`. The cache-hit vs cache-miss boundary is the single fastest way to spot "cache action restores files but skips state" bugs (see ffmpeg/libblas in run 25825814166).
- **[gotcha]** ALL `gh` subcommands (`gh api`, `gh pr create`, `gh run view`, etc.) fail with `Bad credentials (HTTP 401)` when `GITHUB_TOKEN` is set in the environment to an invalid/expired token — `gh` prefers the env var over its stored keyring auth and never falls back. Tell: a tight loop of `gh api repos/<x>/<y>` calls all returning `"status": "401"` consecutively. Fix: `unset GITHUB_TOKEN && gh ...` once at the start of any session that uses `gh`. Already established for `gh pr create`; applies equally to read-only API calls.

- **[pattern]** When fetching a directory tree of source files from a non-GitHub forge (Gitea/Forgejo/sourcehut) for research, the raw-URL pattern differs per host: Gitea/Forgejo use `https://<host>/<owner>/<repo>/raw/branch/<branch>/<path>` (try both `master` and `main`), GitHub uses `raw.githubusercontent.com/<owner>/<repo>/<branch>/<path>`. - **[gotcha]** Research artifacts dumped under `/tmp/` (e.g. `/tmp/dash-research/src/...`) are VOLATILE — wiped on reboot. If a plan references these paths for later phases (citing `Mirrors <repo>/<file>:<line>`), persist the dump alongside the plan: either `cp -r /tmp/<dump> plans/` (gitignored) or stash a tarball, or document the re-fetch commands inside the plan so a future session can rebuild the dump idempotently.
- **[gotcha]** GitHub Actions matrix fan-out is NOT a free speedup — each matrix cell pays the full runner-boot + `actions/checkout@v5 fetch-depth: 0` cost (~30-60s for a monorepo with thousands of commits), so a job that took 30s serial-per-lib can take LONGER as a matrix because every cell re-clones the full history. Before splitting a serial loop into `strategy.matrix`, time one iteration: if per-lib work < ~45s, the per-cell checkout overhead dominates and matrix fan-out is a wash or regression. The `sync-sugarcraft.yml` parallelization (commit ae017e75) had to be reverted for this reason. Alternatives that DON'T pay per-cell checkout: GNU Parallel inside a single job (`echo "$LIBS" | xargs -P 5 -I{} ./sync-one.sh {}`), or a shallow `fetch-depth: 50` if the script only needs recent history.

- **[gotcha]** PHP FFI's `\FFI::cdef()` eagerly resolves ALL declared symbols at library load time — declaring a function the target libc doesn't export (e.g., POSIX 2024 `tcsetwinsize`/`tcgetwinsize` on glibc < 2.36, which is most non-bleeding-edge Linux distros) causes the entire FFI handle to fail with `Failed resolving C function '<name>'`, breaking ALL subsequent FFI calls — not just the missing one. Fix: build the cdef block per-platform — gate POSIX-2024-or-later / Darwin-only / Linux-only declarations behind `if (PHP_OS_FAMILY === 'Darwin')` (or equivalent) when concatenating the cdef string. Canonical pattern: `candy-pty/src/Libc.php::cdef()`. Bare cross-platform declarations only work for symbols guaranteed in every supported libc back to the oldest CI target.
- **[gotcha]** macOS arm64 + PHP FFI + variadic libc functions = ABI mismatch. Real libc `ioctl(int, unsigned long, ...)` is variadic, and the arm64 variadic ABI puts varargs on the stack while fixed args sit in `x0`–`x7`. A fixed-arg PHP FFI cdef (`int ioctl(int fd, unsigned long request, void *arg)`) pushes the third arg into `x2` while the kernel reads it from the stack → ioctl returns `-1` with no useful errno. Workaround: route through non-variadic POSIX 2024 wrappers (`tcsetwinsize`/`tcgetwinsize` for TIOCSWINSZ/TIOCGWINSZ) on Darwin only. PHP FFI cannot reliably express variadic signatures — declaring `int ioctl(int, unsigned long, ...)` breaks Linux too because FFI rejects `FFI\CData` arrays passed as variadic params. There is no portable PHP-FFI-only fix for arbitrary variadic libc calls on macOS arm64.
