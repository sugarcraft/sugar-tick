# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** When a new lib subdir is added, the caliber pre-commit hook does NOT auto-stage untracked files in the new dir. Stage every new file explicitly: `git add candy-<slug>/composer.json candy-<slug>/phpunit.xml candy-<slug>/README.md candy-<slug>/CALIBER_LEARNINGS.md candy-<slug>/src/...` plus the cross-cut edits. `git status -uall <subdir>/` enumerates the per-file list when you forget what you created.
- **[gotcha]** Streaming-writer classes whose `close()` method calls `fclose($this->fh)` (e.g. `candy-vcr`'s `Recorder::close()`) make the underlying stream resource invalid — calling `rewind($fh)` afterwards in the same test throws `TypeError: rewind(): supplied resource is not a valid stream resource`. Test pattern: do NOT call `$r->close()` if you also want to `rewind`/`fgets`/`stream_get_contents` the same `php://memory` handle to inspect the bytes.
- **[pattern]** Cross-lib hook seams that need to live in candy-core but are implemented in a downstream lib (e.g. recorder lives in candy-vcr but the tee point is `Program::writeOutput()` in candy-core) take the form: candy-core ships an INTERFACE, `Program::withRecorder(?Recorder $r): self` accepts the interface, and the downstream lib provides the concrete class implementing it. Avoids candy-core depending on candy-vcr (the dependency goes the other way) and lets other consumers (debug tap, network mirror, crash bundle) slot into the same hook without pulling cassette plumbing.
- **[gotcha]** PSR-4 autoload requires one class per file with basename = class name. Putting test helper/fixture classes at the bottom of a `FooTest.php` works ONLY because PHPUnit `require`s `FooTest.php` to instantiate the test, side-effect-loading the inline classes. If a SECOND test file references those fixtures and runs first in isolation (e.g. `phpunit --filter BarTest`), autoload fails. Move shared fixtures into their own PSR-4-conformant files (`tests/Msg/UserJsonableMsg.php`) from the start when more than one test will use them. Same trap: defining a `final class TickModel` at the bottom of `tests/Assert/ScreenRoundTripTest.php` collides at autoload time if `tests/PlayerTest.php` also defines its own `TickModel` — both files declare the same FQCN even though each is local to one test, causing `Cannot declare class … because the name is already in use` when both load in the same suite. Either use a different per-file class name or extract the fixture to a single shared file.
- **[pattern]** For a `Registry`-of-pluggable-strategies design, use unqualified class names as the routing tag for builtins (readable cassettes/dumps) and FQCN for the catch-all that handles user types. The two coexist if the builtin's `canDecode()` checks against an explicit allowlist (`classForTag()` match expression) so it never accidentally claims a user FQCN. Catch-all serializer (`JsonableSerializer`) round-trips via `new $class(...$data)` named-arg unpacking — works when constructor parameter names match the keys returned by `jsonSerialize()`.
- **[gotcha]** `candy-core`'s `setupTerminal()` emits output bytes (e.g. `\x1b[?2027h` unicodeOn, alt-screen-enter, cursor-hide) BEFORE the startup `WindowSizeMsg` dispatch, and `teardownTerminal()` emits closing counterparts AFTER `dispatch(QuitMsg)` once `loop->stop()` has fired. A `Recorder` (or any `writeOutput` tee) MUST stay open across teardown — closing it on `QuitMsg` drops those teardown bytes from the cassette and then a fresh `Program` on replay emits them, producing a byte-mismatch at the tail. Close the recorder at the END of `Program::run()`, not inside the `QuitMsg` branch.
- **[gotcha]** A `candy-vcr` `Player` driving a `Program` in `SPEED_INSTANT` mode must NOT schedule every cassette event at `delay=0` via `addTimer` — React's `StreamSelectLoop` fires them all in a single tick before the program's framerate-based render tick gets a turn, so `view()` is never rendered. Schedule events sequentially with `Player::INSTANT_YIELD_SECONDS` (~5 ms) between events; in `SPEED_REALTIME` use recorded `t` deltas. Replay programs need a high `framerate` (e.g. 1000.0 in `ProgramOptions`) so the renderer wakes up on the yield interval rather than the default ~60 Hz.
- **[gotcha]** A `candy-vcr` round-trip test (record session A → replay into fresh Program B → assert byte-equal) is NOT achievable with `ByteAssertion` alone — recording captures whatever the framerate-based renderer emitted during the recording loop's lifespan, the replay loop runs over a different chained-timer timeline (Player yields between events to let render fire) and emits a *cell-grid equivalent* but byte-different stream. For PR5 use `ScreenAssertion` (cell-grid via candy-vt) — feeds both byte streams into separate `Terminal` instances and compares grapheme grids via `Screen::diff()`, which tolerates ANSI-level reordering (redundant SGR re-emission, equivalent cursor-position sequences, partial vs full repaints).
- **[gotcha]** When a Player replays a recorded cassette into a FRESH `Program`, both the cassette AND the replay program emit a startup `WindowSizeMsg` (cassette's first `resize` event was the original tty.size() lookup; the new program does its own tty.size() lookup at `setupTerminal()` time). Dispatching the cassette's first resize on replay duplicates the msg, throws the model's tick count off by one, and produces a single-cell screen diff that looks like flake but is deterministic (e.g. `(0,6) expected '6' actual '7'` for a counter-incrementing model). Skip the first `resize` event on replay (`$firstResizeSkipped` flag) — subsequent resizes from in-session SIGWINCH events still need to be dispatched.
- **[gotcha]** When threading a new `&$flag` reference variable through nested closures in PHP, you must add it to EVERY `use()` clause along the path — adding `&$firstResizeSkipped` to the inner `dispatchEvent` closure but forgetting the outer `$step = function () use (...)` closure produces `TypeError: Argument #N (...) must be of type bool, null given` at runtime, NOT a compile-time error. The variable resolves to `null` in the outer scope and gets passed down. Audit every nested `use(...)` when adding a closure-captured variable; `grep -n "use (" Player.php` lists them all in one pass.
- **[pattern]** When a `ByteAssertion`-style diff helper produces a hex window starting at the first-divergence offset (not at offset 0), tests asserting on the diff string cannot expect the FULL hex of either expected/actual — the window only contains the divergent tail. Force the divergence to offset 0 (e.g. `str_repeat('A', 16)` vs `str_repeat('B', 16)`) when you want to assert the full 16-byte hex window.

The class throws `\RuntimeException` if `Terminal::class` doesn't exist, which is the explicit fallback signal to use `ByteAssertion` instead.

- **[gotcha]** The Bash tool's working directory persists across separate Bash calls. After `cd /home/sites/sugarcraft/<lib> && cmd`, the NEXT Bash call still starts in `<lib>` — `git status` will then show paths with `../` prefixes that look like they're outside the repo but aren't. A bare `ls candy-core/src/` from inside `candy-core` returns NO output (empty `{}` response from the tool, not a `No such file or directory` error) because the path is relative to the wrong cwd. Bare `grep candy-core/src/Cmd.php` after `cd .../candy-vcr` produces `ugrep: warning: candy-core/src/Cmd.php: No such file or directory`. Worse, a compound command like `git status --short && git add foo bar baz && git status --short` can return an EMPTY `{}` response (no stdout, no stderr, no exit-code error) when the cwd has drifted to a worktree state where the chained command finds nothing. Always anchor with `cd /home/sites/sugarcraft && ...` to start from the repo root, or pass absolute paths to every command. Same trap bites HEREDOC-written `/tmp/dbg.php` scripts: `require_once __DIR__ . '/vendor/autoload.php'` resolves `__DIR__` to `/tmp/`, NOT to the working dir of the heredoc-emitting Bash call. Use absolute autoload paths (`require_once '/home/sites/sugarcraft/<lib>/vendor/autoload.php';`) in any standalone PHP script you stage under `/tmp/`.

- **[pattern]** For YAML cassettes / human-readable serialization formats backing a value-object (e.g. `Cassette` with header + event list), use `Symfony\Component\Yaml\Yaml::dump($tree, inline: 3, indent: 4)` — `inline: 3` flows the per-event maps onto one line each (compact, diff-friendly) while keeping `header:` and `events:` at the top level on their own lines. Both YAML and JSONL formats should round timestamps the same way (e.g. `round($t, 3)` for ms precision) so cross-format equality holds via the shared value object — tests should explicitly assert this with a `testCrossFormatJsonlYamlRoundTrip` that decodes the same `Cassette` through both formats and compares per-event `t`/`kind`/`payload` triples.

- **[gotcha]** A test helper method named `run()` on a `PHPUnit\Framework\TestCase` subclass dies at autoload with `PHP Fatal error: Cannot override final method PHPUnit\Framework\TestCase::run()` — `TestCase::run()` is final in PHPUnit 10. Pick any other name (`exec()`, `invoke()`, `dispatch()`) for helpers that drive the system-under-test; the same trap also applies to other final TestCase methods like `runBare()` and `runTest()`. Symptom looks like a compile error in the test file itself, not a PHPUnit upgrade issue.
- **[pattern]** PHP `bin/` shim scripts that need to work both from the lib's own `vendor/` (development) AND when composer-installed under a parent project's `vendor/bin/` (the bin shim copy) should iterate two candidate autoload paths: `__DIR__ . '/../vendor/autoload.php'` then `__DIR__ . '/../../../autoload.php'`. The first wins in the lib's own checkout; the second wins when composer installs the bin under `vendor/<vendor>/<pkg>/bin/<script>` and `vendor/autoload.php` is three levels up. Then `composer.json` declares `"bin": ["bin/<script>"]` so composer auto-symlinks it into the consumer's `vendor/bin/`.
- **[gotcha]** A PHP `bin/` shim must be `chmod +x` in the working tree before staging — `git add` preserves the mode bit, but creating the file via the Write tool does NOT mark it executable. Run `chmod +x candy-vcr/bin/<script>` immediately after Write so the committed mode is `100755`; otherwise composer's bin-symlink will install a non-executable file and `vendor/bin/<script>` will fail with `Permission denied`.

- **[gotcha]** A Bash chain like `ls foo 2>/dev/null && echo "---" && ls bar` returns an EMPTY `{}` response from the Bash tool when the first command exits non-zero (e.g. `ls candy-pty` when the dir doesn't exist) — `&&` short-circuits, the entire rest of the chain is suppressed, and the tool surface gives back nothing rather than an error. Recovery: wrap the maybe-failing probe in `( ... || echo "<sentinel>" )` so the chain continues regardless: `(ls -la candy-pty 2>/dev/null || echo "candy-pty does not exist") && echo "---" && ls plans/`. Or use `;` instead of `&&` for unrelated checks. Same root cause as the cwd-drift empty-`{}` trap — any non-zero exit early in a `&&` chain produces a silent black-hole response.

If the gate isn't satisfied, surface that to the user before scaffolding instead of charging ahead.

- **[pattern]** For FFI-dependent test classes (e.g. `candy-pty`'s `Pty::open()` round-trip), do NOT put the FFI/`/dev/ptmx`/PHP_OS_FAMILY skip checks in `setUp()` — that skips the entire class on FFI-less CI runners and you lose coverage on structural tests (cdef shape, env-override resolution, exception hierarchy) that don't actually touch the syscalls. Instead extract a private `requirePtySyscalls()` (or similarly named) helper and call it as the FIRST line of each test that opens a real pty. Structural tests run unconditionally; round-trip tests skip cleanly when the host can't support them.
