# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) ŌĆö do not edit manually.

- **[gotcha]** Cross-cut tracking files exist for some upstream ports but NOT others ŌĆö `plans/x-windows.md` (the `charmbracelet/x/windows` port) was tracked only in `plans/` and `candy-core/`, with NO entry in `MATCHUPS.md`, `UPSTREAM_OPPORTUNITIES.md`, `docs/index.html`, or any per-lib README until the implementation shipped. Plans for sub-modules of `charmbracelet/x` don't always graduate to a standalone lib row ŌĆö they ship as additions to existing libs (e.g. `candy-core/src/Util/Tty/`). Don't assume a `plans/x-<feature>.md` file implies a new MATCHUPS row; check whether the work modifies an existing lib in-place before adding tracking entries.
- **[pattern]** The `plans/x-<feature>.md` file's `## Tracking` section enumerates the post-implementation sweep ŌĆö files outside the implementation PRs that need updates after the LAST PR merges (UPSTREAM_OPPORTUNITIES status icon, root CALIBER_LEARNINGS gotchas, README/AGENTS platform requirements). Treat this as a checklist; do a final tracking-only PR (e.g. `ai/<feature>-tracking`) when all implementation PRs land, since these doc updates don't fit into any individual implementation PR's scope.
- **[gotcha]** Each lib's `phpunit.xml` names its testsuite after the lib (`<testsuite name="candy-vt">`, `name="candy-core"`, etc.) ŌĆö there is NO `default` testsuite. `vendor/bin/phpunit --testsuite default` prints `No tests executed!` and exits 0. Run `vendor/bin/phpunit` with no `--testsuite` flag, or pass the actual lib name; if you must filter, `grep -m1 'testsuite name' phpunit.xml` confirms the right value.
- **[pattern]** When porting a state-machine table or other binary-encoded structure from upstream Go, mirror the enum numeric values EXACTLY (e.g. `enum State: int { case Ground = 0; case CsiEntry = 1; ... }` matching upstream `iota` order). Lookup keys like `state.value << 8 | byte` are bit-compatible only when the ordering is preserved ŌĆö silent off-by-one mismatches produce confusing test failures hundreds of bytes deep. Add a doc-comment noting "Order is load-bearing ŌĆö transition table indexed by state value."
- **[pattern]** To grab a specific upstream Charmbracelet file for offline reference (when porting), use `gh api repos/<owner>/<repo>/contents/<path>/<file> --jq '.content' | base64 -d > /tmp/<file>` rather than cloning the whole repo. The contents API returns base64-encoded blobs; loop over a list of filenames in a for-loop. `gh api repos/<owner>/<repo>/contents/<dir> --jq '.[] | .name'` lists the dir first.
- **[gotcha]** Upstream charmbracelet/x/ansi/parser's transition table executes CAN (0x18) and SUB (0x1A) anywhere with `Action::Execute ŌåÆ Ground` ŌĆö they do NOT silently cancel the in-flight sequence. Tests for "CAN cancels CSI" must assert both the `execute` event for the cancellation byte AND the print/print-like event for the byte that follows. Same applies to many C1 controls in the Anywhere block.
- **[gotcha]** When committing back-to-back with the caliber hook active, `git commit` can fail with `fatal: cannot lock ref 'HEAD': is at <new-sha> but expected <old-sha>` after printing `caliber: skipping finalize ŌĆö another finalize is in progress`. The commit object DOES land (the `<new-sha>` in the error is the freshly-created commit) ŌĆö caliber's background finalize from a previous commit was still running and lost the race to update HEAD. Before retrying, run `git log -1 --oneline && git status` ŌĆö if HEAD already points at the SHA from the error and the tree is clean, the commit succeeded; a retry would fail with "nothing to commit". Don't `git commit --amend` either ŌĆö there's nothing to amend.
- **[pattern]** When swapping a stub method (e.g. `Terminal::feed()` no-op ŌåÆ real Parser+Handler pipeline) for a working implementation, audit the test file FIRST and replace the corresponding stub-era test (`testFeedIsNoOpInPr1` etc.) before running the suite ŌĆö leaving the stub-era assertion in place produces a confusing failure that looks like the new wiring is broken ("expected ' ' got 'R'") when it's actually the test that's outdated. PHPUnit's diff output shows the new behavior on the `+` side, which is a hint to update the test rather than the code.
- **[pattern]** When writing tests that drive a VT parser/handler with raw byte sequences, set up buffer state with explicit CSI cursor-position moves (`\x1b[r;cHTEXT`) rather than chains of print + CR (`\x0D`) + LF (`\x0A`) + overwrite. Mental-tracing the latter is error-prone ŌĆö `testLinefeedAtBottomScrollsAndStays` and `testCsiSScrollsUp` both failed on first run because the author miscounted what landed where after `"A\x0DB\x0AC\x0DD\x0AE"`-style sequences with overwriting at the cursor; rewriting them as `\x1b[1;1HAB\x1b[2;1HCD\x0A` (explicit row,col placement, then exactly one LF to trigger the behavior under test) made the assertions trivially correct. Reserve C0/CR/LF chains for tests that are *specifically* about CR/LF semantics.
- **[gotcha]** When `git commit` returns `nothing to commit, working tree clean` immediately after producing a real change-set, suspect that caliber's pre-commit hook from a *previous* commit ran the finalize asynchronously and already committed your staged files. Run `git log -1 --oneline && git status` ŌĆö if the most recent commit is the one you intended, you're done; do NOT re-stage and re-commit (you'll get an empty/duplicate commit or a HEAD-lock race).
- **[convention]** Mode field names in `candy-vt`'s `Mode` class don't align 1:1 with DEC private mode numbers ŌĆö e.g. `mouseAny` Ōåö 1000 (X11 button-only), `mouseExtended` Ōåö 1003 (any-motion), `mouseSgr` Ōåö 1006 (SGR coords). Don't infer the mapping from field names; check `ModeHandler::applyOne()`'s `match` table or `candy-vt/CALIBER_LEARNINGS.md`'s mapping table when wiring a new DEC mode. `Mode::$mouseHighlights` is reserved for 1001 but unwired ŌĆö reading it as "highlights mode is on" is wrong.
- **[pattern]** When a `ScreenHandler::<oscDispatch|csiDispatch|...>()` slot still has a `// PR<N> placeholder` no-op comment, that's the wiring point for the matching sub-handler. Replace the body with `$this->fooHandler->apply($data, $this);` and add the sub-handler instance to the constructor ŌĆö don't introduce a new dispatch layer. Sub-handlers (Sgr/Cursor/Erase/Scroll/Mode/Osc) follow the same shape: stateless class with `apply($params, ScreenHandler)` writing into public slots on the handler.
- **[gotcha]** The Bash tool's working directory persists across separate Bash calls. After `cd /home/sites/SugarCraft/sugar-charts && cmd`, the NEXT Bash call still starts in sugar-charts. Always use absolute paths or prefix every Bash call with `cd /home/sites/SugarCraft && ...` to anchor it; if you get a mysterious empty response from `ls`/`cat`, suspect the cwd before suspecting the file. Same trap bites `php -r "file_get_contents(__DIR__.'/tests/fixtures/foo')"` ŌĆö `__DIR__` in `-r` mode resolves to the cwd at exec time, so a stale cwd from an earlier `cd tests/fixtures` produces `tests/fixtures/tests/fixtures/foo` and a `Failed to open stream` warning. Use absolute paths in `php -r` snippets.
- **[pattern]** When swapping a stub method (e.g. `Terminal::feed()` no-op ŌåÆ real Parser+Handler pipeline) for a working implementation, audit the test file FIRST and replace the corresponding stub-era test (`testFeedIsNoOpInPr1`, `testPrintsMultiByteUtf8AsSingleCell`, etc.) before running the suite ŌĆö leaving the stub-era assertion in place produces a confusing failure that looks like the new wiring is broken ("expected 1 got 2" cell count for CJK after Width wiring) when it's actually the test that's outdated. PHPUnit's diff output shows the new behavior on the `+` side, which is a hint to update the test rather than the code.
- **[pattern]** Binary test fixtures (raw ANSI/control byte streams under `tests/fixtures/*.ansi`) need a dir-local `.gitattributes` declaring `*.ansi binary` ŌĆö without it, git's `core.autocrlf` may rewrite `\r\n` Ōåö `\n` on checkout and silently corrupt CR/LF/ESC byte sequences in the fixture. Generate the bytes via `printf '\x1b[31mR\x1b[0m' > tests/fixtures/foo.ansi` rather than an editor (which may insert/strip BOMs or normalise newlines). Verify with `bin2hex(file_get_contents(...))` from a `php -r` one-liner using an absolute path.
- **[pattern]** When auditing whether a newly-added library is FULLY integrated, the easy-to-miss artifacts are (a) `media/icons/<slug>.png` AND `docs/img/icons/<slug>.png` (BOTH paths ŌĆö `media/` is the source-of-truth icon, `docs/img/icons/` is what the website serves), (b) the `docs/index.html` homepage tile in the `.logo-strip` section (`<a href="lib/<slug>.html" title="<Brand> ŌĆö <one-liner>"><img src="img/icons/<slug>.png" alt="<Brand>"></a>`), (c) the `docs/lib/<slug>.html` detail page (use `docs/lib/candy-core.html` as the reference layout ŌĆö head/og meta, nav, `.detail-header` with icon + h2 + tag-chips, install/quickstart/feature-grid/source/demos sections, footer), (d) `<slug>/examples/*.php` runnable demos, (e) `<slug>/.vhs/*.tape` + matrix entry in `.github/workflows/vhs.yml`. Nothing fails without them ŌĆö composer/CI/tests all pass ŌĆö so they linger. Audit checklist: `ls media/icons/<slug>.png docs/img/icons/<slug>.png docs/lib/<slug>.html <slug>/examples <slug>/.vhs && grep -q '<slug>' docs/index.html .github/workflows/vhs.yml`.
- **[pattern]** When a new lib needs a placeholder icon, two interchangeable approaches: (a) ImageMagick `convert -size 256x256 xc:transparent -fill '<hex>' -draw "..."  <path>.png`, or (b) PHP GD via `php -r '$im = imagecreatetruecolor(256, 256); ...; imagepng($im, $path);'` (works without ImageMagick). Write to BOTH `media/icons/<slug>.png` and `docs/img/icons/<slug>.png` in the same script. Verify with `file <path>` ŌĆö should report `PNG image data, 256 x 256`. Existing icons are 240┬▓ but the contributor playbook calls for 256┬▓ ŌĆö both render fine in `docs/index.html`'s `.logo-strip` (CSS resizes), so don't waste time matching the legacy 240┬▓ dimension.
- **[gotcha]** After `gh pr create` / `gh pr merge --delete-branch`, the script prints `Warning: N uncommitted changes` (any small N) even on a clean tree ŌĆö this is `gh` complaining about untracked files like `.claude/scheduled_tasks.lock` or sibling `plans/` files left over from another branch's work, not a real uncommitted-source warning. Don't react to it; verify with `git status` if unsure.

The cell-grid form survives ANSI-level reordering ŌĆö redundant SGR re-emission, equivalent cursor moves ŌĆö that doesn't change what the user sees. - **[gotcha]** `end()`, `reset()`, `next()`, `prev()`, `array_pop()`, `array_shift()`, `sort()`, etc. all take their argument by reference and CANNOT be called on a `public readonly array $foo` property ŌĆö you get `Error: Cannot modify readonly property` pointing at the line that calls the function (not at any assignment), which is misleading. Use index access (`$this->foo[count($this->foo) - 1]`) or `array_key_last($this->foo)` instead. This bites both `final readonly class` properties and per-property `public readonly array` declarations equally ŌĆö the readonly modifier blocks the by-ref pointer mutation that those funcs perform.
- **[gotcha]** When a new lib subdir is added (e.g. `candy-vcr/`), the caliber pre-commit hook does NOT auto-stage untracked files in the new dir ŌĆö `git add -A` would, but per project convention you stage explicit paths to avoid sweeping in `.env`-class files. Stage every new file explicitly: `git add candy-<slug>/composer.json candy-<slug>/phpunit.xml candy-<slug>/README.md candy-<slug>/CALIBER_LEARNINGS.md candy-<slug>/src/...` plus the cross-cut edits (`composer.json` root, `.github/workflows/ci.yml`, `MATCHUPS.md`, `PROJECT_NAMES.md`, `CONVERSION.md`, `UPSTREAM_OPPORTUNITIES.md`). `git status -uall <subdir>/` enumerates the per-file list when you forget what you created.
- **[pattern]** When wiring a new lib into the root `composer.json`, BOTH the `require` block (add `"sugarcraft/<slug>": "@dev"`) AND the `repositories` block (add `{"type":"path","url":"<slug>","options":{"symlink":true}}`) need entries ŌĆö missing the repositories block produces `Could not find package` at install time even though the directory exists alongside. Pair the two edits in the same step.
