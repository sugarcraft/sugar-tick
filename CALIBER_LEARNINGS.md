# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** `Color::rgb($r, $g, $b)` throws `InvalidArgumentException: rgb component out of range [0,255]` for any component > 255. When synthesising test pixel grids with arithmetic like `Color::rgb(64 * $c, 64 * $r, 128)`, wrap with `min(255, ...)` or pick a step that can't exceed 255.
- **[gotcha]** `echo` and `exit` are PHP language constructs, NOT functions — so you cannot define `function echo(...)` (parse error) and you cannot call `\exit(main($argv))` (runtime `Call to undefined function exit()`). Both forms compile-/run-fail and cannot be "namespaced". When porting older code or using these in `match` arms (`'help' => echo(...)`), wrap output in a real helper named `say()`/`out()` and use bare `exit(main($argv))` (no leading backslash). Run `php -l <file>` on every CLI bin you touch.
- **[correction]** Monorepo-wide contributor docs (locale codes, naming policy, conversion roadmap) belong at the REPO ROOT alongside `MATCHUPS.md`/`CONVERSION.md`/`PROJECT_NAMES.md`, NOT under a specific lib's directory like `candy-core/lang/`. **Why:** every lib uses these codes when contributors add `lang/<locale>.php` files, and burying the doc inside `candy-core/` leaks contributor-only guidance into the downstream-distributed Packagist package. **How to apply:** when creating a new cross-cut reference doc, place it at root and add it to `AGENTS.md`'s "Source-of-truth files" list.
- **[gotcha]** From a `.claude/worktrees/<name>/` worktree, `git checkout master` fails with `fatal: 'master' is already used by worktree at '/home/sites/sugarcraft'`. Don't try to switch — `git fetch origin master --quiet && git reset --hard origin/master` to update the current detached HEAD, then `git checkout -b claude/<feature>` to start a fresh branch. Or work in a non-master branch from the start.
- **[gotcha]** `vendor/bin/phpunit` from repo root fails with `No such file or directory` — the root `composer.json` doesn't necessarily get a `composer install` run, only per-lib directories do. Always `cd <lib> && (composer install --quiet &&) vendor/bin/phpunit`, or wrap in a subshell loop iterating the canonical lib list. Don't assume root has a `vendor/` populated.
- **[pattern]** When auditing whether a newly-added library is FULLY integrated, the easy-to-miss artifacts are `media/icons/<slug>.png` (256² candy-themed icon) and the `docs/index.html` homepage tile (`<a href="lib/<slug>.html" title="<Brand> — <one-liner>"><img src="img/icons/<slug>.png" alt="<Brand>"></a>`). Nothing fails without them — composer/CI/tests all pass — so they linger. Audit checklist: `ls media/icons/<slug>.png && grep -q "<slug>" docs/index.html` to confirm both exist before declaring the lib integrated.
- **[pattern]** When a new lib needs an icon and no fixture exists, synthesize a placeholder via ImageMagick: `convert -size 256x256 xc:transparent -fill '<hex>' -draw "rectangle x1,y1 x2,y2" ... -blur 0x3 media/icons/<slug>.png`. Existing icons are 240² but the contributor playbook calls for 256² — both render fine in `docs/index.html`'s `.logo-strip` (CSS resizes), so don't waste time matching the legacy 240² dimension.
- **[gotcha]** Updating MATCHUPS.md / UPSTREAM_OPPORTUNITIES.md status icons (🟡 → 🟢) is a separate audit step from the per-PR ship cycle — `gh pr merge` doesn't update them. After the LAST PR in a multi-PR plan merges, grep both files for the upstream slug and bump the status icon, otherwise the public-site grids and roadmap docs stay stuck at "in progress" indefinitely.
- **[gotcha]** `git commit` with the caliber pre-commit hook active (`grep -q "caliber" .git/hooks/pre-commit`) frequently exceeds the default 120s Bash timeout while `caliber refresh` regenerates CLAUDE.md / AGENTS.md / .claude/. If the Bash call gets killed mid-hook, the commit never records (HEAD unchanged) but caliber's edits to CLAUDE.md/AGENTS.md remain in the working tree — looks like a partial state. Pass `timeout: 600000` to the Bash tool when committing alongside changes that touch caliber-managed sections.
- **[gotcha]** Cross-cut tracking files exist for some upstream ports but NOT others — `plans/x-windows.md` (the `charmbracelet/x/windows` port) is tracked only in `plans/` and `candy-core/`, with NO entry in `MATCHUPS.md`, `UPSTREAM_OPPORTUNITIES.md`, `docs/index.html`, or any per-lib README. Plans for sub-modules of `charmbracelet/x` don't always graduate to a standalone lib row — they ship as additions to existing libs (e.g. `candy-core/src/Util/Tty/`). Don't assume a `plans/x-<feature>.md` file implies a new MATCHUPS row; check whether the work modifies an existing lib in-place before adding tracking entries.
- **[pattern]** Cross-platform CI matrices are split into TWO jobs in `.github/workflows/ci.yml`: the main `test` job (Linux, runs every lib) and a separate `windows-test` job (windows-latest, runs only TTY-sensitive libs: `candy-core sugar-prompt sugar-bits candy-shell candy-shine`). Composer cache is intentionally DISABLED on the Windows job due to GH Actions PowerShell path-encoding issues — leave it that way; `vendor/` caching handles the meaningful layer. When adding TTY-sensitive code paths, add the lib to the windows-test matrix too, not just the main matrix.
- **[gotcha]** Cross-cut tracking files exist for some upstream ports but NOT others — `plans/x-windows.md` (the `charmbracelet/x/windows` port) is tracked only in `plans/` and `candy-core/`, with NO entry in `MATCHUPS.md`, `UPSTREAM_OPPORTUNITIES.md`, `docs/index.html`, or any per-lib README until the implementation ships. Plans for sub-modules of `charmbracelet/x` don't always graduate to a standalone lib row — they ship as additions to existing libs (e.g. `candy-core/src/Util/Tty/`). Don't assume a `plans/x-<feature>.md` file implies a new MATCHUPS row; check whether the work modifies an existing lib in-place before adding tracking entries.
- **[pattern]** When a port introduces NEW platform/runtime requirements (PHP 8.4+ for Windows FFI, Windows 10 1809+ for `ENABLE_VIRTUAL_TERMINAL_PROCESSING`, an extension like `ffi`), update BOTH the per-lib README's `## Requirements` section (concrete extension/version list) AND `AGENTS.md`'s opening-line monorepo summary (high-level constraint visible to every contributor). Touching only one leaves contributors with conflicting expectations — README readers learn the gotcha, AGENTS-only readers don't. Check `grep -n 'PHP 8' AGENTS.md candy-core/README.md` after the implementation PR merges.
- **[gotcha]** After `gh pr create` / `gh pr merge --delete-branch`, the script prints `Warning: N uncommitted changes` (any small N) even on a clean tree — this is `gh` complaining about the lockfile diff that `composer install` left behind, or untracked sibling files like `plans/` or `.tmp/` from a different branch's work, not a real uncommitted-source warning. Don't react to it; verify with `git status` if unsure.
- **[gotcha:windows-ffi]** Windows FFI to kernel32.dll requires PHP 8.4+ for `SetConsoleCtrlHandler` — the callback closure must be kept alive as a static class property for the entire process lifetime; if GC'd, Windows calls into freed memory. Use a static property on `WindowsBackend` to hold the closure reference. Also: with PHPUnit `failOnWarning=true`, an `E_USER_WARNING` emitted from a `PHP < 8.4` graceful-fallback branch will abort the whole Windows CI suite — make the fallback truly silent (return false without `trigger_error`).
- **[gotcha]** Cross-cut tracking files exist for some upstream ports but NOT others — `plans/x-windows.md` (the `charmbracelet/x/windows` port) was tracked only in `plans/` and `candy-core/`, with NO entry in `MATCHUPS.md`, `UPSTREAM_OPPORTUNITIES.md`, `docs/index.html`, or any per-lib README until the implementation shipped. Plans for sub-modules of `charmbracelet/x` don't always graduate to a standalone lib row — they ship as additions to existing libs (e.g. `candy-core/src/Util/Tty/`). Don't assume a `plans/x-<feature>.md` file implies a new MATCHUPS row; check whether the work modifies an existing lib in-place before adding tracking entries.
- **[pattern]** The `plans/x-<feature>.md` file's `## Tracking` section enumerates the post-implementation sweep — files outside the implementation PRs that need updates after the LAST PR merges (UPSTREAM_OPPORTUNITIES status icon, root CALIBER_LEARNINGS gotchas, README/AGENTS platform requirements). Treat this as a checklist; do a final tracking-only PR (e.g. `ai/<feature>-tracking`) when all implementation PRs land, since these doc updates don't fit into any individual implementation PR's scope.
- **[gotcha]** Each lib's `phpunit.xml` names its testsuite after the lib (`<testsuite name="candy-vt">`, `name="candy-core"`, etc.) — there is NO `default` testsuite. `vendor/bin/phpunit --testsuite default` prints `No tests executed!` and exits 0. Run `vendor/bin/phpunit` with no `--testsuite` flag, or pass the actual lib name; if you must filter, `grep -m1 'testsuite name' phpunit.xml` confirms the right value.
- **[pattern]** When porting a state-machine table or other binary-encoded structure from upstream Go, mirror the enum numeric values EXACTLY (e.g. `enum State: int { case Ground = 0; case CsiEntry = 1; ... }` matching upstream `iota` order). Lookup keys like `state.value << 8 | byte` are bit-compatible only when the ordering is preserved — silent off-by-one mismatches produce confusing test failures hundreds of bytes deep. Add a doc-comment noting "Order is load-bearing — transition table indexed by state value."
- **[pattern]** To grab a specific upstream Charmbracelet file for offline reference (when porting), use `gh api repos/<owner>/<repo>/contents/<path>/<file> --jq '.content' | base64 -d > /tmp/<file>` rather than cloning the whole repo. The contents API returns base64-encoded blobs; loop over a list of filenames in a for-loop. `gh api repos/<owner>/<repo>/contents/<dir> --jq '.[] | .name'` lists the dir first.
- **[gotcha]** Upstream charmbracelet/x/ansi/parser's transition table executes CAN (0x18) and SUB (0x1A) anywhere with `Action::Execute → Ground` — they do NOT silently cancel the in-flight sequence. Tests for "CAN cancels CSI" must assert both the `execute` event for the cancellation byte AND the print/print-like event for the byte that follows. Same applies to many C1 controls in the Anywhere block.
- **[gotcha]** When committing back-to-back with the caliber hook active, `git commit` can fail with `fatal: cannot lock ref 'HEAD': is at <new-sha> but expected <old-sha>` after printing `caliber: skipping finalize — another finalize is in progress`. The commit object DOES land (the `<new-sha>` in the error is the freshly-created commit) — caliber's background finalize from a previous commit was still running and lost the race to update HEAD. Before retrying, run `git log -1 --oneline && git status` — if HEAD already points at the SHA from the error and the tree is clean, the commit succeeded; a retry would fail with "nothing to commit". Don't `git commit --amend` either — there's nothing to amend.
- **[pattern]** When swapping a stub method (e.g. `Terminal::feed()` no-op → real Parser+Handler pipeline) for a working implementation, audit the test file FIRST and replace the corresponding stub-era test (`testFeedIsNoOpInPr1` etc.) before running the suite — leaving the stub-era assertion in place produces a confusing failure that looks like the new wiring is broken ("expected ' ' got 'R'") when it's actually the test that's outdated. PHPUnit's diff output shows the new behavior on the `+` side, which is a hint to update the test rather than the code.
- **[pattern]** When writing tests that drive a VT parser/handler with raw byte sequences, set up buffer state with explicit CSI cursor-position moves (`\x1b[r;cHTEXT`) rather than chains of print + CR (`\x0D`) + LF (`\x0A`) + overwrite. Mental-tracing the latter is error-prone — `testLinefeedAtBottomScrollsAndStays` and `testCsiSScrollsUp` both failed on first run because the author miscounted what landed where after `"A\x0DB\x0AC\x0DD\x0AE"`-style sequences with overwriting at the cursor; rewriting them as `\x1b[1;1HAB\x1b[2;1HCD\x0A` (explicit row,col placement, then exactly one LF to trigger the behavior under test) made the assertions trivially correct. Reserve C0/CR/LF chains for tests that are *specifically* about CR/LF semantics.
- **[gotcha]** When `git commit` returns `nothing to commit, working tree clean` immediately after producing a real change-set, suspect that caliber's pre-commit hook from a *previous* commit ran the finalize asynchronously and already committed your staged files. Run `git log -1 --oneline && git status` — if the most recent commit is the one you intended, you're done; do NOT re-stage and re-commit (you'll get an empty/duplicate commit or a HEAD-lock race).
- **[convention]** Mode field names in `candy-vt`'s `Mode` class don't align 1:1 with DEC private mode numbers — e.g. `mouseAny` ↔ 1000 (X11 button-only), `mouseExtended` ↔ 1003 (any-motion), `mouseSgr` ↔ 1006 (SGR coords). Don't infer the mapping from field names; check `ModeHandler::applyOne()`'s `match` table or `candy-vt/CALIBER_LEARNINGS.md`'s mapping table when wiring a new DEC mode. `Mode::$mouseHighlights` is reserved for 1001 but unwired — reading it as "highlights mode is on" is wrong.
- **[pattern]** When a `ScreenHandler::<oscDispatch|csiDispatch|...>()` slot still has a `// PR<N> placeholder` no-op comment, that's the wiring point for the matching sub-handler. Replace the body with `$this->fooHandler->apply($data, $this);` and add the sub-handler instance to the constructor — don't introduce a new dispatch layer. Sub-handlers (Sgr/Cursor/Erase/Scroll/Mode/Osc) follow the same shape: stateless class with `apply($params, ScreenHandler)` writing into public slots on the handler.
- **[gotcha]** The Bash tool's working directory persists across separate Bash calls. After `cd /home/sites/SugarCraft/sugar-charts && cmd`, the NEXT Bash call still starts in sugar-charts. Always use absolute paths or prefix every Bash call with `cd /home/sites/SugarCraft && ...` to anchor it; if you get a mysterious empty response from `ls`/`cat`, suspect the cwd before suspecting the file. Same trap bites `php -r "file_get_contents(__DIR__.'/tests/fixtures/foo')"` — `__DIR__` in `-r` mode resolves to the cwd at exec time, so a stale cwd from an earlier `cd tests/fixtures` produces `tests/fixtures/tests/fixtures/foo` and a `Failed to open stream` warning. Use absolute paths in `php -r` snippets.
- **[pattern]** When swapping a stub method (e.g. `Terminal::feed()` no-op → real Parser+Handler pipeline) for a working implementation, audit the test file FIRST and replace the corresponding stub-era test (`testFeedIsNoOpInPr1`, `testPrintsMultiByteUtf8AsSingleCell`, etc.) before running the suite — leaving the stub-era assertion in place produces a confusing failure that looks like the new wiring is broken ("expected 1 got 2" cell count for CJK after Width wiring) when it's actually the test that's outdated. PHPUnit's diff output shows the new behavior on the `+` side, which is a hint to update the test rather than the code.
- **[pattern]** Binary test fixtures (raw ANSI/control byte streams under `tests/fixtures/*.ansi`) need a dir-local `.gitattributes` declaring `*.ansi binary` — without it, git's `core.autocrlf` may rewrite `\r\n` ↔ `\n` on checkout and silently corrupt CR/LF/ESC byte sequences in the fixture. Generate the bytes via `printf '\x1b[31mR\x1b[0m' > tests/fixtures/foo.ansi` rather than an editor (which may insert/strip BOMs or normalise newlines). Verify with `bin2hex(file_get_contents(...))` from a `php -r` one-liner using an absolute path.
