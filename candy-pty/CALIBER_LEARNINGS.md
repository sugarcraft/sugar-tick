# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern:php-shim-vs-fork]** Two paths to "claim the slave as the child's ctty between fork and exec" exist: (a) PHP shim invoked via proc_open (chosen for PR7), or (b) FFI fork+exec inside the parent's PHP runtime. (a) wins for SugarCraft because PHP's runtime is fork-hostile — zval allocator, opcache, atexit handlers, FFI handles all carry over. The shim approach inherits proc_open's clean fork semantics: child immediately exec's PHP→pcntl_exec'd cmd, no PHP user code runs in the forked child. Shim startup cost (~5-50 ms) is the trade — accept it, opt the path in only when callers ask for ctty (`controllingTerminal: true`). Use `PHP_BINARY` as the interpreter so shim runs with the same PHP version as the parent.
- **[gotcha:ioctl-third-arg-ulong-not-pointer]** `ioctl()`'s third arg is read by the kernel as `unsigned long`, not as a dereference of a pointer. For TIOCSCTTY where we want `arg=0` ("don't steal an existing ctty"), passing PHP `null` to the FFI cdef declaration `int ioctl(int fd, unsigned long request, void *arg)` works — null pointer renders as 0 in the unsigned-long reinterpret. Avoid the temptation to allocate `$libc->new('int')` with `$arg->cdata = 0` and pass `FFI::addr($arg)` — that would pass a pointer-to-zero, NOT zero, and the kernel reinterprets it as a non-zero pointer address. Same `void *arg` signature handles TIOCSWINSZ (real pointer to winsize struct) cleanly — both arg types coexist in one cdef.
- **[pattern:test-sigwinch-self-delivery]** Smoke-testing SIGWINCH-driven resize logic without a real terminal: use `posix_kill(posix_getpid(), SIGWINCH)` to deliver SIGWINCH to the current process. Combined with a closure-bound size provider (`fn () => ['cols' => $cols, 'rows' => $rows]` capturing mutable locals by reference), tests can mutate the dims and trigger a self-SIGWINCH to verify the handler picks up the new size and ioctls TIOCSWINSZ on the master fd. See `candy-pty/examples/resize-forwarding.php`.
- **[pattern:macos-ci-scope]** The macOS CI job in `.github/workflows/ci.yml` is intentionally scoped to a single lib (`candy-pty`) rather than the full matrix — Darwin runners are slow and most libs don't have platform-specific code paths. When adding a new lib that touches libc FFI / TIOCSWINSZ-style ioctls / `posix_*` syscalls with platform-divergent constants, append it to the `macos-test` matrix. The job mirrors the `windows-test` block's structure (concurrency group with `ci-macos-` prefix, vendor cache key with `vendor-mac-` prefix, same shivammathur/setup-php@v2 setup minus the FFI-on-Windows quirks).
- **[gotcha:tiocsctty]** PR2 wires the slave path to `proc_open()`'s `[0,1,2]` descriptor slots — opening `/dev/pts/N` three times — so the kernel hands the child stdio that talks to the master. But the child does NOT claim the slave as its **controlling terminal** (no `setsid()` + `ioctl(slave_fd, TIOCSCTTY, 0)`). Consequence: `Ctrl+C` typed into the master does NOT generate a `SIGINT` for the child — the kernel has nowhere to deliver job-control signals. Tests using `/bin/true`, `/bin/false`, `sleep`, `sh -c 'exit N'` are unaffected (no signal-aware logic). Anything interactive (vim, less, bash with line-editing) will mis-handle Ctrl+C. PR7 ships the fix as an opt-in shim — pass `controllingTerminal: true` to `Pty::spawn()` and the spawn is routed through `bin/pty-shim.php` which does `setsid()` + `ioctl(0, TIOCSCTTY, 0)` + `pcntl_exec()`.
- **[pattern:tioscctty-shim]** To claim a slave PTY as a child's controlling terminal from PHP, you cannot do it inline in `proc_open` — the kernel only honours `TIOCSCTTY` from a session leader, and `proc_open` doesn't `setsid()`. Pattern that works: ship a small PHP shim (`bin/pty-shim.php` with `#!/usr/bin/env php` hashbang and `chmod +x`) and prepend `[PHP_BINARY, /path/to/shim.php]` to the cmd. The shim does `setsid()` (via FFI to libc) → `ioctl(0, TIOCSCTTY, 0)` → `pcntl_exec(cmd, args)` between `proc_open` and the actual cmd. Costs ~5-50ms of shim startup so make it opt-in (default off). Canonical implementation in `candy-pty/bin/pty-shim.php` + `Spawn::wrapInShim()`.
- **[gotcha:tioscctty-constants]** The `TIOCSCTTY` ioctl request number is platform-divergent — Linux uses `0x540E`, macOS uses `0x20007461` (the `_IOWR('t', 1, int)` macro encoding). Branch on `PHP_OS_FAMILY === 'Darwin'`. The third arg is read by the kernel as `unsigned long`; passing PHP `null` to an FFI `void *` param renders as 0 which the kernel interprets as "don't steal an existing ctty from another session." Same pattern applies to other `_IOxx`-encoded ioctl numbers when porting POSIX code touching FFI.
- **[pattern:test-tioscctty-pid-sid]** Smoke-test "did `setsid()` land?" without needing a real terminal: spawn `/bin/sh -c "echo $$ > tmp; ps -o sid= -p $$ >> tmp"` via the shim, then assert `pid == sid` in the captured file. If the child became its own session leader, the shell's `$$` matches the kernel's reported `sid`. Compare against the same spawn without `controllingTerminal:true` — pid will NOT equal sid because the child inherits the parent's session.
- **[pattern:test-ctrl-c-sigint]** Smoke-test "does Ctrl+C deliver SIGINT through TIOCSCTTY?" — spawn `/bin/sleep 10` with `controllingTerminal:true`, `usleep(150_000)` to let `sleep` settle in its syscall, write `"\x03"` to the master, then `wait()`. Elapsed time should be well under 2s (kill was immediate) rather than the full 10s. Inverse test with `controllingTerminal:false` + `/bin/sleep 0.5` should run to completion (Ctrl+C goes nowhere) — confirms the shim is what makes the difference.
- **[pattern:default-libc-path-match]** When adding a new bin/* or aux script that loads libc via FFI, mirror `Libc::libraryPath()` exactly: honour the `SUGARCRAFT_LIBC` env var override, default to `libc.so.6` on Linux and `/usr/lib/libSystem.B.dylib` on macOS. Diverging defaults across the codebase causes hard-to-debug load failures on musl/Alpine where users override once and expect every entry-point to pick it up.
- **[pattern:opaque-termios-struct]** — struct termios layout differs glibc/musl/Darwin; treat as opaque buffer, only call cfmakeraw/tcgetattr/tcsetattr.
- **[pattern:wait-loop-with-drain]** PosixProcess::wait() must drain captured stdout/stderr pipes per poll iteration or `yes | head`-style children deadlock on a full kernel pipe buffer while the parent sleeps in `usleep(10_000)`. Trait's plain wait() doesn't do this — PosixProcess overrides it explicitly with a `drainCapturePipes()` call inside the loop.
- **[antipattern:final-extend]** Don't `extends` a class marked `final` — fatal at autoload, no migration path. P2.5 MasterAdapter was the post-mortem: needed to compose a `PosixMasterPty` (which is `final`) inside a `MasterAdapter` to bridge legacy Pty facade callers, so the adapter held the master as a member and forwarded calls rather than subclassing.
- **[pattern:posix-kill-zero-probe]** `posix_kill($pid, 0)` returns false + ESRCH cleanly when the pid no longer exists; use it for liveness checks after init reaped the child. Beats parsing `/proc/<pid>/status` (Linux-only) or reading `proc_get_status()` on a process you don't own.
- **[pattern:pump-no-wait]** `PosixPump::run()` deliberately does NOT call `$child->wait()` when the child is still alive — returns -1 instead. Caller (InProcessTransport, candy-shell) owns the kill-on-stdin-EOF policy: pump returning -1 means "I gave the child stdinEofGraceSec to notice, it didn't, your move." Inverting would let a daemon-style child hold the pump hostage for its full lifetime.
- **[pattern:shim-startup-budget]** Controlling-terminal shim (`bin/pty-shim.php`) costs ~5–50 ms of PHP boot + FFI cdef. Always opt-in via `controllingTerminal: true`, never default-on. Non-interactive children (`echo`, `tput`, `bash -c 'one-liner'`) don't need ctty so they bypass the shim and stay snappy.
- **[pattern:no-throw-in-signal-handler]** Every callback delivered via `SignalForwarder::attachSigwinch` / `attachSigchld` is wrapped in `try { } catch (\Throwable) { }`. PHP signal handlers run between opcodes — a thrown exception unwinds across arbitrary user code with a corrupt stack. The pattern is non-negotiable for any new signal-driven feature.
- **[pattern:stdin-eof-grace]** PosixPump's STDIN-EOF handling: on stdin EOF, write VEOF (`\x04`) to master, then start a `stdinEofGraceSec` (default 300 ms) timer. If child exits in the window, pump returns child's exit code; if it doesn't, pump returns -1 and caller force-closes master (delivers SIGHUP via lost-ctty). Both arms are tested in P5.5 `EofGraceTest`.
- **[antipattern:seekable-stdin-drift]** Tests that pass a seekable stream (e.g. `php://memory`) as stdin to PosixPump trigger silent drift: `fread()` doesn't return EOF when the position reaches end-of-data because new writes can extend it. Use `php://temp` or a fresh pipe via `stream_socket_pair()` for tests; canonical fix in `PumpEofTest::makePipe()`.
- **[pattern:isclosed-on-contract]** `MasterPty::isClosed()` is on the contract (not just the implementation) so signal handlers / supervisors can guard `resize()` / `write()` against a torn-down session without depending on a concrete class. The SIGWINCH handler short-circuits on `isClosed()` to dodge the SIGWINCH-arrives-during-close race.
- **[pattern:sigint-settle-window]** Smoke-testing SIGINT delivery: `usleep(150_000)` after spawn before writing `\x03` to the master. Without the settle window the shell hasn't reached its read() syscall yet and the kernel queues the SIGINT for nothing — the child eats the byte as input. 150 ms is comfortable; 50 ms is flaky.
- **[pattern:path-repo-closure]** Every consuming lib's composer.json must list the FULL transitive path-repo closure, not just direct deps. Candy-wish depends on candy-pty which depends on candy-core — candy-wish's `repositories[]` needs all three. Diagnosis: `composer install` fails with "Could not find a matching version of package sugarcraft/<dep>" because Composer doesn't recursively resolve `type: path` repos.
- **[pattern:cdef-symbol-frugality]** Libc cdef block is intentionally narrow — only the symbols a feature needs land in the cdef on first load. Adds a symbol per PR rather than a maximalist all-of-termios block at PR1 so any cdef syntax error surfaces against a small surface, easier to bisect. Same rule for libc.cdef() in `Libc::cdef()`.
- **[gotcha:macos-tcsetwinsize-missing]** macOS 15 (Sequoia) libSystem does NOT ship POSIX 2024 `tcsetwinsize` / `tcgetwinsize` despite their availability being documented for macOS 13+. FFI cdef-load fails eagerly with `Failed resolving C function 'tcsetwinsize'` on first `FFI::cdef()` call — even if call sites are platform-gated. Do NOT add these symbols to the libc cdef on any platform; use `ioctl(TIOCSWINSZ)` with a `stty -f /dev/fd/<fd>` shell fallback for the Darwin arm64 variadic ABI mismatch. Re-evaluate when macOS 16+ ships them.
- **[gotcha:ffi-eager-symbol-resolve]** `FFI::cdef()` resolves ALL declared symbols against the loaded library at cdef time — declaring a symbol the library doesn't export throws even if no code ever calls it. Branching cdef text per platform (`if (PHP_OS_FAMILY === 'Darwin')`) is necessary but NOT sufficient — verify the symbol actually exists on the OLDEST target OS version (e.g. the CI runner's libSystem, not your dev box) before adding it. Wrapping the call in `try/catch` won't help; the throw happens at cdef load, before any user code runs.
- **[gotcha:composer-update-lock-flag]** `composer update <pkg> --lock --no-install` fails with `You cannot simultaneously update only a selection of packages and regenerate the lock file metadata.` Drop the `--lock` flag — `composer update <pkg> --no-install` already writes the lock file and skips `vendor/` install. The `--lock`-only mode is for `composer update --lock` with no package arg (refresh metadata).
- **[gotcha:dependent-lock-files-on-branch]** When a downstream lib (e.g. `candy-pty`) switches to a feature branch in its consuming lib's `composer.json` via implicit `dev-master` → `dev-<branch>` resolution, every consuming lib's `composer.lock` still pinned to `dev-master` will fail CI with `Your lock file does not contain a compatible set of packages. sugarcraft/<dep> is locked to version dev-master and an update of this package was not requested.` Fix: after pushing the feature branch, run `cd <consuming-lib> && composer update sugarcraft/<dep> --no-install --no-interaction` to refresh its lock to the new branch SHA, and commit the lock change in the same PR.
- **[gotcha:ioctl-read-vs-write-variadic]** On macOS arm64 our fixed-arg FFI `ioctl(int, unsigned long, void *)` cdef bug surfaces ONLY in the write direction with payload-bearing structs — `ioctl(TIOCSWINSZ)` fails (struct pointer to wrong register, kernel returns -1), but `ioctl(TIOCGWINSZ)` works (read direction, kernel writes to the buffer through a path that tolerates the register layout). Don't assume "if SET broke, GET also needs the fallback" — measure each direction independently. Canonical asymmetry in `SizeIoctl::setSizeViaLibc` (stty fallback) vs `SizeIoctl::getSizeViaLibc` (plain ioctl).
- **[gotcha:recorder-header-defaults]** `Recorder::open($path)` without a second arg writes a header with the **default** cols/rows (80×24) regardless of any `recordResize($cols, $rows)` event you emit later — `recordResize` only lands an event row, it doesn't backfill the header. Downstream readers that trust `cassette.header->cols`/`rows` (e.g. `JsonlFormat::read()`) will see the defaults. To honour custom dims, call `Recorder::open($path, Recorder::defaultHeader($cols, $rows, 'sugarcraft/<lib>@<command>'))`. Caught by `RecordCommandTest::testCustomColsRowsLandInCassetteHeader` asserting `header->cols === 132` and failing with `80 is identical to 132`.
 - **[pattern:bash-watchdog-cd-isolation]** When wrapping a PTY-touching phpunit run with a `( sleep N && pkill ... ) &` watchdog, the `cd <lib> && (sleep ... ) & WATCHDOG_PID=$! && vendor/bin/phpunit` chain races: each Bash tool call starts a fresh shell, and `cd` BEFORE the backgrounded subshell doesn't reliably anchor the subsequent `vendor/bin/phpunit` invocation. Repeat the `cd <abs-path>` on BOTH the watchdog branch AND the phpunit branch, or use absolute paths for the phpunit binary. The CWD-doesn't-persist gotcha from CLAUDE.md applies inside this chain too — observed in a candy-pty full-suite run that landed in `/home/sites/sugarcraft/candy-vcr` instead.
 - **[pattern:onIdle-vs-onSigwinch]** Step 01.03 split the idle tick into a distinct `onIdle` hook. `onIdle` fires on every `stream_select` timeout (no I/O ready) — use it for keepalive, polling, periodic housekeeping. `onSigwinch` fires only when terminal dimensions change — never pass `(0, 0)` to fake an idle tick; use `withOnIdle()` instead. `InProcessTransport` wires its `keepaliveCallback` → `PumpOptions::onIdle` directly since step 01.03.
 - **[pattern:deferred-backend-exception]** — UnsupportedPlatformException::forDeferredBackend($backend) throws with "deferred to phase 12" message; phase-12 implementers must wire the backend AND remove the throw
 - **[gotcha:fopen-fd-dup-close-after-fclose]** `fopen('php://fd/N')` dup()s the fd (php-src plain_wrapper.c) — `fclose` only closes the dup, not the original. Every `posix_openpt` master fd must be closed explicitly via libc `close()` after any `fclose($stream)` or the kernel's master-side refcount never reaches 0 and `tty_hangup()` never fires (no SIGHUP for session leader). Diagnosed in step 01.12 via SignalForwarderReactLoop test timeout (>2 s) — child survived because fd leak kept master refcount alive.
 - **[pattern:fd-cloexec-on-master]** Every `posix_openpt` master fd must have `FD_CLOEXEC` set before `proc_open` — otherwise the forked child inherits the master fd, keeping the kernel-side refcount > 0 after parent close, blocking `tty_hangup()`. `fcntl($masterFd, F_SETFD, FD_CLOEXEC)` right after `posix_openpt` is required. Same applies to any anchor slave fd held across the fork boundary (macOS). Step 01.12 surfaced this as the second half of the SIGHUP delivery bug.
 - **[pattern:openpty-darwin-first]** Darwin's `openpty(3)` is a single-call alternative to the `posix_openpt + grantpt + unlockpt + ptsname_r` quartet. In `PosixPtySystem::openPtyMaster()`, Darwin attempts `openpty()` first; on -1 it falls through to the quartet. The slave fd returned by `openpty()` is closed immediately — only the master is needed in the parent. The child gets stdio via `proc_open`. This mirrors the portable-pty / creack/pty preference for `openpty()` on macOS where it is the canonical path.
 - **[pattern:openpty-libutil-linux]** On Linux, `openpty(3)` lives in `libutil.so.1` (not libc) on most distributions. The FFI loader tries `libc.so.6` first (via `Libc::libraryPath()` honouring `SUGARCRAFT_LIBC`), then falls back to `libutil.so.1` specifically for the `openpty` symbol. Darwin ships `openpty` in libSystem.B.dylib so no fallback is needed there. If a Linux system lacks `libutil.so.1` (rare, e.g. some musl builds), `openpty` is unavailable and the quartet is used instead.
 - **[pattern:waitpid-ffi-fast-path]** `ChildPollTrait::tryWaitpid()` uses `waitpid(pid, &status, WNOHANG)` via FFI as a sub-ms exit-detection fast-path before falling back to `proc_get_status`. WNOHANG=1 means non-blocking — returns 0 if child still running, >0 (the pid) if exited. Exit code decoding: normally-exited → `(status >> 8) & 0xFF`; signal-terminated → `128 + (status & 0x7F)` (Unix convention). The FFI call is wrapped in `try/catch` so unavailability of `ext-ffi` or the `waitpid` symbol degrades gracefully to the poll loop. The signal-exit convention (128+signal) mirrors what shells do — `exit 2` from a signal handler yields `130` (2+128).
