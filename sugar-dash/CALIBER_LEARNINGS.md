# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha:project]** sugar-dash declares secondary value-object classes (e.g. `AreaPoint`, `BubblePoint`, `ConsoleStream`, `ConsoleEntry`) at the top of the same file as their owning class (`Area.php`, `Bubble.php`, `Console.php`). PSR-4 autoloads by class name → file name, so `use SugarCraft\Dash\Grid\ConsoleStream` from a test file fails with `Class "SugarCraft\Dash\Grid\ConsoleStream" not found` even though the class is defined — Composer never opens `Console.php` looking for `ConsoleStream`. Fix: extract each secondary class into its own PSR-4-matching file (`src/Grid/ConsoleStream.php`, `src/Grid/ConsoleEntry.php`, etc.) and remove the inline declaration from the owning file. When adding any non-trivial secondary class (enum or class) to a sugar-dash Grid file, give it its own file from the start.
- **[gotcha:project]** `Canvas::drawCircle()` uses Bresenham's midpoint algorithm and draws only the perimeter — it does NOT set the center pixel `($cx, $cy)`. `getPixel(10, 10)` after `drawCircle(10, 10, 5)` returns `null`. Either explicitly set the center in `drawCircle()`, use `fillCircle()` instead, or adjust tests that assert the center is colored. Tests in `tests/Grid/CanvasTest.php` should assert cardinal-point pixels (e.g. `(10, 5)`, `(10, 15)`, `(5, 10)`, `(15, 10)` for a radius-5 circle at `(10, 10)`), not the center.
- **[gotcha:project]** sugar-dash `ClassDiagram::render()` (and similar diagrams) historically guarded with `|| empty($this->classes)` and returned `''`, which makes `testRenderReturnsNonEmptyString` fail for a fresh `ClassDiagram::new()->setSize(70, 20)->render()` with no classes added. If a test asserts a non-empty render on a default instance, remove the empty-collection short-circuit and let the render produce at least the chrome (title bar, borders). Apply the same audit to any other `Grid/*.php` `render()` that bails on `empty($this->items)`.
- **[gotcha:project]** Render-with-color tests on `Canvas` (and other ANSI-emitting renderers) can't `assertSame` a row's plain-text width once colored — the ANSI escape bytes inflate the string. Use `assertStringContainsString('     █', $lines[1])` rather than `assertSame('     █     ', rtrim($lines[1]))`, or strip ANSI with the project's `Ansi::strip()` helper before comparing.
- **[gotcha:project]** sugar-dash `Console` keeps DUAL state: `private array $entries` (filtered, view cache) AND `private array $originalEntries` (source of truth). Most withers reconstruct via `new self(entries: $this->originalEntries, ...)` which means any wither that ADDS/REMOVES entries MUST mutate `originalEntries`, not the filtered `entries` array. Symptom of the bug: `Console::new()->withEntry($e)->withShowPrefix(false)->render()` returns empty string because `withShowPrefix` rebuilt from `originalEntries` (still empty) and dropped the entry added to `entries`. Pattern: in `withEntry()` do `$clone->originalEntries = array_merge($clone->originalEntries, [$entry]); $clone->entries = $this->filterEntries($clone->originalEntries);`. Apply this audit to any class with both filtered and unfiltered collections.
- **[gotcha:project]** `assertSame('#F9FAFB', strtolower($actual))` ALWAYS fails because the expected literal stays uppercase while the actual is lowercased — the comparison is asymmetric. Either pre-lowercase the literal (`assertSame(strtolower('#F9FAFB'), strtolower($actual))`) or compare uppercase both sides. This bites whenever `Color::toHex()` returns lowercase but the test author copy-pasted hex codes in their canonical uppercase form. Grep `tests/Grid/` for `assertSame\('#[A-F]` to find latent failures of this form.

- **[pattern:project]** When extracting an inline secondary class to its own PSR-4 file (e.g. `OHLCPoint` out of `OHLC.php`), do TWO edits: (1) `Write` the new `src/Grid/<Name>.php` with the full class body plus `<?php`, `declare(strict_types=1);`, `namespace SugarCraft\Dash\Grid;`, and any `use` statements the class needs (esp. `SugarCraft\Core\Util\Color`); (2) `Edit` the original file to delete the inline class declaration including its preceding docblock, leaving only the owning class. - **[gotcha:project]** sugar-dash chart label-row rendering must compute chars-per-column from the chart width: `$perPoint = max(1, intdiv($chartWidth, max(1, $dataCount)))` then `mb_substr($label, 0, $perPoint)` + `str_pad($label, $perPoint)`. The naive `mb_substr($label, 0, 1)` or hardcoded `mb_substr($label, 0, 2)` collapses every label to one or two characters, so `assertStringContainsString('Jan', $rendered)` or `assertStringContainsString('AAPL', $rendered)` fails with only `J` or `AA` visible. Same trap exists in `Area::render()`, `Bubble::render()`, `OHLC::render()`, and any other chart with X-axis labels — search `grep -n "mb_substr.*->label" src/Grid/*.php` when chart text tests fail. For OHLC-style charts where each point gets a `$barWidth + $spacing` cell, use `$cellWidth = $barWidth + $spacing` as the label truncation length, not the narrower `$barWidth`.
- **[gotcha:project]** Removing an empty-collection short-circuit guard from a sugar-dash `render()` (e.g. dropping `|| empty($this->states)` so a fresh `State::new()->setSize(60,20)->render()` returns chrome) frequently surfaces a `DivisionByZeroError` from arithmetic further down that assumed >0 elements: `intval(sqrt($stateCount))`, `intval(($stateCount + $cols - 1) / $cols)`, `intval(($width - 4) / $cols)`, `intval($chartHeight / $count)`. After removing the outer guard, audit the body and wrap every `count`-derived divisor with `max(1, ...)` (e.g. `intval(sqrt(max(1, $stateCount)))`, `max(1, intval(...))`). Run `grep -nE 'intval\([^)]*/ ?\$' src/Grid/<file>.php` to catch them.
- **[gotcha:project]** sugar-dash `render()` methods often have BOTH an outer width/height/empty guard AND an inner `if ($count === 0) return '';` inside the private `renderDiagram()` helper (seen in `Sequence.php`, likely also `Class*`, `PERT`, `Sankey`). - **[gotcha:project]** sugar-dash chart components with a fixed default-height fallback (`$this->height ?? 20`) silently clip the trailing items when collection size exceeds the canvas. `Sankey::render()` with `useHeight = 20` and three default-spacing nodes drops the third — `testAddMultipleNodes` fails on `assertStringContainsString('Node 3', $rendered)`. Default height MUST be derived from collection size: `$this->height ?? max(20, count($this->nodes) * ($this->nodeSpacing + 5) + 8)`. Same trap exists in any chart where rendered-row count scales with collection size — audit every `?? 20` / `?? 10` / `?? 15` literal in `Grid/*.php` `render()` methods and replace with a `max(default, count(...) * row_height)` formula.
- **[gotcha:project]** Treemap-style cell layout that advances `$currentX += $cellWidth + 1;` (with a one-column gap between cells) overflows the available width and silently drops subsequent leaves — e.g. two leaves of width 23+23 in a 47-wide canvas exit early because `currentX = 24` followed by another `cellWidth = 23` triggers `currentX + cellWidth > useWidth` and the second leaf wraps to a row that's then `break`-ed out by the height check. Cells must butt directly against each other: `$currentX += $cellWidth;` with no `+1`. Also bump `max(2, intval(...))` to `max(3, intval(...))` so labels have room to render. The same bug pattern lurks in any `Grid/*.php` layout loop that uses both an inter-cell offset AND a wrap-on-overflow check.
- **[pattern:project]** When a Treemap-style cell renders multiple fill rows and needs a centered label, compute `$labelRow = ($this->showLabels && $innerHeight > 0) ? intval($innerHeight / 2) : -1;` outside the loop and only emit the label when `$row === $labelRow` — `mb_substr($label, 0, $innerWidth)` with `intval($padding/2)` left pad and `$padding - $leftPad` right pad. Computing `$density` / `$fillIndex` / `$fillChar` outside the loop avoids repeating it per-row.
- **[gotcha:project]** Tests that slice a multi-line ANSI render with `$lines = explode("\n", $render); $lines[1]` then assert `assertStringContainsString('█', $stripped)` are fragile: if a fix changes the thumb / content position, the assertion locks to the wrong row. When the test is really asserting "this rendering contains a thumb glyph", strip ANSI from the FULL render and `assertStringContainsString('█', $stripped)` — don't index into a specific line unless the test name implies a positional contract. Grep `tests/Grid/` for `explode\("\\n".*\[\d+\]` to find latent failures of this form.
