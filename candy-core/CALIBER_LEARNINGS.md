# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha:project]** `gh` commands fail silently (empty `{}` response from Bash) when `$GITHUB_TOKEN` is set to an invalid token, even though `~/.config/gh/hosts.yml` has a working cached token. `gh auth status` confirms it: "X Failed to log in to github.com using token (GITHUB_TOKEN)" + "✓ Logged in to github.com account ... (hosts.yml)". Workaround: prefix every `gh` invocation with `unset GITHUB_TOKEN &&`. Do NOT try to overwrite `GITHUB_TOKEN` with a fresh PAT inline — `gh auth login --with-token` rejects new tokens here too ("Bad credentials").
- **[env:project]** This is a multi-package monorepo where each library (`candy-core/`, `candy-shine/`, `sugar-bits/`, etc.) has its own `composer.json` + `vendor/`. The umbrella `/home/sites/SugarCraft/composer.json` is `type: metapackage` and does NOT install phpunit. To run tests for a library, `cd <lib>/ && composer install` first; the resulting `<lib>/vendor/bin/phpunit` is what runs the suite. Don't look for phpunit in the parent `vendor/bin/`.
- **[gotcha:project]** `WebFetch` against `raw.githubusercontent.com/<owner>/<repo>/<branch>/<path>` returns an empty `{}` (no error, no message) when the path doesn't exist. Seen with guessed filenames like `bubblezone/master/zone.go` and `bubblezone/master/info.go` (real names are `zoneinfo.go` / `manager_global.go` / `scanner.go`). Recovery: fetch the GitHub directory listing page (`https://github.com/<owner>/<repo>`) first to confirm filenames, then re-fetch the raw URL. Do not retry the same path — the empty response means "file not found," not transient.
- **[convention:project]** Library dependency order for sequential fixes is Phase 0 (`candy-core/src/Util/`) → Phase 1 (`candy-sprinkles/`) → Phase 2 (`honey-bounce/`) → Phase 3 (`candy-core/`) → Phase 4 (`candy-zone/`) → Phase 9 (`candy-shine/`) → Phase 5 (`sugar-bits/`) → Phase 6 (`sugar-charts/`) → Phase 7 (`sugar-prompt/`) → Phase 8 (`candy-shell/`). Phases 9 and 5+ depend on Phase 0/1/3 utilities (OSC 8, `Width::wrap`, `Exec`), so foundation fixes must land first.
- **[pattern:project]** When auditing PHP ports against Go upstreams (`charmbracelet/*`, `lrstanley/bubblezone`, `NimbleMarkets/ntcharts`), use `WebFetch` on the raw `.go` source file with a prompt like "List every exported function/method on the Foo Model. One-line signatures." Returns a clean public-API surface in ~3-5s per file. Cross-reference against the PHP `src/` to surface missing methods. Faster than reading the full Go source.
- **[pattern:project]** Testing the `Renderer` (or any class that writes to a `php://memory` resource across multiple calls) — DO NOT reset the stream between writes with `ftruncate($out, 0); rewind($out);`. The renderer holds no position handle of its own, but the underlying stream's write pointer matters and `ftruncate` followed by another write produced empty reads in this session. Instead, let the stream grow and slice deltas: `$end = ftell($out); $r->render(...); fseek($out, $end); $delta = stream_get_contents($out);`. This is the canonical pattern in `candy-core/tests/RendererTest.php`.

- **[pattern:catalog]** Named-constant classes (spinner styles, themes, color palettes) should expose a static `catalog(): list<string>` method returning all available names. This enables discovery and iteration without hardcoding counts. Pattern: `public static function catalog(): array { return ['item1', 'item2', ...]; }`. Applied to `SugarCraft\Bits\Spinner\Style`, `SugarCraft\Prompt\Theme`, and `SugarCraft\Palette\StandardColors`.

- **[pattern:stream-helper]** Extract stream-write test logic into a `StreamHelper` trait. The canonical pattern: save position with `ftell()`, execute write, `fflush()`, seek back with `fseek()`, read delta with `stream_get_contents()`. Use `createTempStream()` to get a `php://memory` stream for renderer tests. Located at `candy-core/src/Test/StreamHelper.php`.
- **[gotcha:project]** When unit-testing a diff algorithm with a "fall back to full repaint when partial would be larger" branch (cell-diff Renderer), plain unstyled inputs like `'AB' → 'XY'` will NOT trigger the fallback — the partial path's payload is just `cursorTo + eraseToLineEnd + 'XY'` and partial wins. To exercise the fallback path you need inputs whose partial repaint must re-emit SGR state to be correct (e.g. `\x1b[1;31mAlpha\x1b[0m` → `\x1b[1;31mXY\x1b[0m`); the SGR re-emission overhead is what makes the full repaint shorter.

- **[gotcha:windows-ffi]** `SetConsoleCtrlHandler` via FFI requires PHP 8.4's `FFI::dynamicFunction()` to safely cast a PHP closure to a C function pointer. On PHP < 8.4 the call throws. In practice the Ctrl-handler path is only needed when `openTty()` succeeded (real Windows console, not headless CI). Use `@trigger_error` to suppress the warning since the method already returns false gracefully on failure.

- **[gotcha:windows-ci]** On `windows-latest` GitHub Actions runner: `putenv('COLUMNS')` does not override the runner-reported terminal size — `GetConsoleScreenBufferInfo` always returns the actual console dimensions (80x24). Tests that set COLUMNS/LINES env vars must skip on Windows CI.

- **[gotcha:windows-ci]** Windows CI runner has pcntl-like signal support via CreateProcess, so `Tty::onResize()` may return `true` on Windows CI while returning `false` in CLI tests (where pcntl is absent). Do not assume pcntl absence means no resize handling — check `DIRECTORY_SEPARATOR === '\\'` first.

- **[gotcha:windows-kernel32]** `CreateFileW('CONIN$')` on the Windows CI runner (headless ConHost) returns a valid handle that passes `GetLastError() == 0`, but `ReadConsoleInputW` on that handle may hang indefinitely. The correct headless-safe pattern: return `null` when `GetLastError() == 6` (`ERROR_INVALID_HANDLE`). Tests can override via `WindowsBackend::setTestKernel32()` to inject a fake that returns fd 0.

- **[gotcha:php-stream-wrapper]** `fopen('php://fd/0', 'rb')` on Windows wraps the raw OS handle for stdin. On headless CI this returns EOF; on a real Windows console it reads the console input buffer. This is the correct pattern for `CONIN$` handles in `WindowsBackend::openTty()`. Do NOT use `php://stdin` — that can be redirected.

- **[gotcha:trigger-warning]** PHPUnit with `failOnWarning=true` in `phpunit.xml` aborts the entire suite on any `E_USER_WARNING`. Every `trigger_error` in production code that may run during suite load must be suppressed with `@trigger_error(...)` to prevent silent suite aborts.

 - **[pattern:async-cmd]** When implementing async commands via `Cmd::promise()` and `AsyncCmd`, rejected promises are automatically converted to `ExceptionMsg` by the `Program::dispatch()` method. However, for consistency with sync backends (which return error messages as `Message::assistant('_[error: ...]_')`), consumers should handle errors in the promise chain itself rather than relying on `ExceptionMsg`. Example: `$backend->completeAsync(...)->then(fn(Message $m) => new AssistantMsg($m), fn(\\Throwable $e) => new AssistantMsg(Message::assistant('_[error: ' . $e->getMessage() . ']_')));`

- **[coverage:kernel32]** `Kernel32` class line coverage is fundamentally limited on Linux. All instance methods call `ffi()` which loads kernel32.dll via FFI - this throws `FFI\Exception` on non-Windows platforms. The only testable surface on Linux is:
  - `self()` static factory (doesn't call `ffi()`)
  - Interface constants defined in `Kernel32Interface` (don't require instantiation)
  - The actual Windows console integration is tested via `FakeKernel32` in `WindowsBackendTest` which provides a test double implementing the interface without FFI.
  - On a real Windows box with kernel32.dll, the full suite could be run for true coverage.

 - **[pattern:interface-default-trait]** PHP interfaces cannot carry default method implementations. When an interface method has a sensible default (e.g., `subscriptions(): null`), provide a trait that implements the default and have concrete classes `use` the trait. This avoids forcing every implementor to repeat the boilerplate. Example: `SubscriptionCapable` trait provides `subscriptions(): null` for `Model` implementors that don't need subscriptions — the trait satisfies the interface contract with zero per-class code.

 - **[pattern:screen-stack:ssot]** `ScreenStack` is the canonical immutable stack type for modal / sub-screen workflows. It is `readonly`/`final`, exposes `push()`/`pop()` returning new instances (never mutates `$this`), plus `current()`/`breadcrumb()`/`isEmpty()`/`count()`. All screen-routing flows in candy-core use this SSOT.

 - **[pattern:screen-stack:screen-vo]** `Screen` is a value object representing one screen in a stack. It carries `model` (the `Model` instance), optional `title` (for breadcrumb display), and two lifecycle closures: `onEnter` (`Closure(): void`) and `onExit` (`Closure(): void`). The closures are nullable — callers invoke them conditionally: `$screen->onEnter?->__invoke()`. The `ScreenStackCapable` interface signals to `Program` that screen-stack routing should be activated.

 - **[pattern:screen-stack:root-model]** `RootModelWithScreenStack` is the concrete root model implementing both `Model` and `ScreenStackCapable`. It handles `ScreenStackPushedMsg` and `ScreenStackPoppedMsg` infrastructure messages in `update()`, invoking lifecycle closures when screens are pushed/popped. The `view()` method delegates to `$this->currentModel?->view() ?? ''`. Consumers can subclass this instead of reimplementing the routing logic from scratch.

 - **[pattern:component-lifecycle]** `Component` interface (`onMount()`/`onUnmount()`) and `Composite` model form a parent-child lifecycle system. `Composite` holds `pendingCmds` (list) and calls `drainCompositePendingCmds()` during reconcile to fire child `onMount()`/`onUnmount()` Cmds. When a `Component` is mounted, `onMount()` returns a `Cmd` (or null); when unmounted, `onUnmount()` returns a `Cmd` (or null). The runtime drains these Cmds synchronously during the same update cycle — no separate scheduling phase. Children are reconciled by ordinal position (`$children[$i]`) not by key, so reordering triggers unmount/mount of affected children. `Program` integrates `Composite` at the top level to drive the root lifecycle.

- Lang class now extends `SugarCraft\Core\I18n\Lang` — `t()` method inherited from base; NAMESPACE and DIR are the only per-lib constants.
