# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern]** Before `gh pr create` / `gh pr merge` in this repo, prefix the command with `unset GITHUB_TOKEN &&` — the workflow used `unset GITHUB_TOKEN && gh pr create ...` and `unset GITHUB_TOKEN && gh pr merge ...` to ensure `gh` falls back to the user's stored auth token rather than a stale env var that may be present in the shell. Without this, `gh` can attach with the wrong identity or fail silently.
- **[gotcha]** `Input` field in sugar-prompt does NOT have `withDefault()` — that method only exists on certain field types. A test added `Input::new('a')->withDefault('A')` and failed with `Call to undefined method`. To set initial values on an `Input`, use `setValue(...)` (or whatever the field-specific setter is); don't assume every Field carries `withDefault`.
- **[pattern]** When adding a long-running child-process feature that needs captured output (e.g. `--show-output`), spawn `RealProcess` with `captureStdout: true` and use non-blocking pipes (`stream_set_blocking($pipe, false)`). Drain the pipes inside `exitCode()` BEFORE checking `proc_get_status`, otherwise the child can deadlock waiting for the parent to read stdout once the OS pipe buffer fills.
- **[pattern:command-attribute-discovery]** `CommandScanner::scan()` uses `get_declared_classes()` to find `#[Command]` classes — it only sees already-loaded classes. Commands must be autoloaded (via Composer) or explicitly `require`d before calling `Application::scan()`. The scanner does not trigger autoloading. Also: `Flag::$enum` holds a class-string (e.g. `FormatType::class`) but Symfony's `InputOption` has no native allowed-values constraint for options (unlike `InputArgument::VALUE_ARGUMENT_MODE`). Full enum validation requires a normalizer or post-processing step in the command's `execute()`; the attribute plumbing is present but the live validation is not yet wired.
- **[pattern:typo-suggestion]** `TypoSuggester` uses Levenshtein distance ≤ 2 to suggest the nearest registered command when a user types a non-existent command name. `Application::find()` wraps `parent::find()` in a try-catch; on `CommandNotFoundException` it calls the suggester against `array_keys($this->all())`. Commands within threshold 0–2 get a "Did you mean `<info>suggestion</info>?" exception; beyond threshold the original exception propagates silently.
- **[pattern:help-formatter-attributes]** `HelpFormatter` discovers `#[Alias]` and `#[Example]` attributes via `ReflectionClass::getAttributes()` — the same pattern as `CommandScanner` for `#[Command]`. `getAttributes()` returns an array sorted by declaration order; accessing `[0]` retrieves the first attribute instance. Both alias and example arrays are iterated with `array_map` to extract `->name` and `->usage`/`->description` respectively.
- **[pattern:env-var-fallback]** `Application::doRun()` applies `CANDYSHELL_*` env var fallbacks via `applyEnvVarFallbackToInput()`. Option names are uppercased with non-alphanumerics replaced by `_`, prefixed with `CANDYSHELL_`. The fallback iterates command options and calls `$input->setOption()` only when `getenv()` returns a value — it does NOT check whether an explicit CLI option was already provided. Explicit options win by virtue of being parsed from `$argv` into `ArgvInput` when the command executes; env vars fill the gap when no explicit value exists. The input must be bound to the command's definition before calling `setOption()`, otherwise Symfony's `Input::setOption()` throws "option does not exist".
- **[pattern:version-from-composer]** `Application::versionFromComposer()` reads the version by first calling `findRootComposer()` which walks up from `__DIR__` via `dirname()` until it finds a `composer.json` with a non-empty `version` field. This allows any sub-package to discover the monorepo root version without hardcoding a path. The root `composer.json` carries the actual `version` field used for `--version` output and the `Application` constructor passes it to `parent::__construct('candyshell', $this->versionFromComposer())`.

- Lang class now extends `SugarCraft\Core\I18n\Lang` — `t()` method inherited from base; NAMESPACE and DIR are the only per-lib constants.

- **[pattern:inverted-dependency-resolved]** candy-shell now depends on `candy-forms` (foundation) instead of `sugar-bits` / `sugar-prompt` (leaf). The inverted dependency from improvements.md §2.1 is resolved: all `use SugarCraft\Bits\*` / `SugarCraft\Prompt\*` imports were repointed to `SugarCraft\Forms\*` (TextInput, TextArea, ItemList, Viewport, FilePicker, Spinner, Confirm all live in candy-forms now), and the `sugarcraft/sugar-bits` + `sugarcraft/sugar-prompt` entries were dropped from both `require` and `repositories`. candy-forms is a foundation lib that does NOT depend on the leaf libs, so the path-repo closure stays complete.
